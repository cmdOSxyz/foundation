# 09.02 Stage 1 (MVP) Build Plan

Status: Active
Owner: Solo build
Horizon: 3–6 months
North star: the full cmdOS specification under `docs/`. This plan implements a thin
vertical slice of that vision — it does not replace or narrow it.

---

## 0. Principle

Stage 1 executes the specification **in dependency order**, exactly as `CLAUDE.md`
prescribes:

```
Core Runtime → Agent System → Capability → Permission → Execution → UI
```

Marketplace, Enterprise, mobile, cloud, and the token layer are **later Stages** (2–4)
and are intentionally out of scope here. They remain in the repo as the destination.

Definition of done for Stage 1: a standalone desktop app where a user types a
natural-language intent and cmdOS completes a **real filesystem/document task** through
the full canonical loop:

```
Intent → Understanding → Command → Execution Plan → Permission → Runtime → Verification → Result
```

reliably (target ≥ 90% success on the supported task set), with every step observable.

---

## 1. The wedge

- **One capability domain:** local filesystem + documents (list, rename/move in bulk,
  convert formats, extract data from PDF/Word).
- **One platform:** desktop (Windows first, macOS second).
- **Why this wedge:** no third-party OAuth, results are verifiable (before/after diff),
  small blast radius, reversible (backup before write), and demos value in seconds.

Stack: Electron + TypeScript + React. One LLM API call performs planning. This maps to
the repo's Interaction / Intelligence / Agent / Capability / Runtime layers without
building the full 54-subsystem kernel yet — a minimal executor + append-only event log
stands in for the kernel during Stage 1.

---

## 2. Milestones

Each milestone is ~1 month (two 2-week sprints). Each maps to a `CLAUDE.md` priority, a
repo chapter, and a segment of the canonical loop, and has a gate that must pass before
the next begins.

### M0 — Foundations & contracts (2 weeks)
Maps to: repo structure (`06.03`), design principles (`03`), "interface first" rule.

- Scaffold the repo per the canonical layout: `apps/desktop/`, `schemas/`,
  `capabilities/`, `kernel/` (minimal), `tests/`.
- Define the core interfaces in `schemas/` (TypeScript / JSON Schema):
  `Intent`, `ExecutionPlan`, `PlanStep`, `Capability`, `PermissionRequest`, `Event`.
- Stand up the append-only **event log** (observability foundation).
- Boot an empty Electron + React shell.

**Gate:** contracts compile; the empty app boots; an event can be written and read back.

### M1 — Core Runtime (1 month)
Maps to: Engineering Priority #1 · kernel `05.7.*` (minimal) · loop: Command → Runtime.

- Minimal executor: takes an `ExecutionPlan`, runs `PlanStep`s in deterministic order.
- Authoritative in-memory/on-disk **state**; immutable **Command** objects.
- Emit an event per step (started/succeeded/failed).

**Gate:** a hardcoded plan (e.g. "rename these 3 files") runs through the runtime, with a
full event trail and correct final state.

### M2 — Agent System + Intelligence (1 month)
Maps to: Engineering Priority #2 · chapters `05.1` (AI) and `05.2` (Agent) · loop:
Intent → Understanding → Execution Plan.

- One LLM call turns a natural-language `Intent` into a structured `ExecutionPlan` (JSON).
- Agent loop: interpret goal → request plan → hand plan to the executor.
- **Semantic validation** of the plan before anything runs (schema + sanity checks).

**Gate:** type an intent in plain language → receive a valid, schema-checked plan
(display only; not yet executed).

### M3 — Capability System + Permission (1 month)
Maps to: Engineering Priority #3 and #4 · chapters `05.3` (Capability) and `06`/security ·
loop: Permission → Runtime.

- `Capability` contract + a small registry. First capability: `filesystem`.
- **Permission gate:** every write/delete/move requires explicit user approval before
  execution (repo rule: sensitive actions need approval).
- Blast-radius control: back up before write; make operations reversible.

**Gate:** a real end-to-end run on the filesystem, gated by an approval dialog, with an
undo path.

### M4 — Execution Engine hardening + Verification (1 month)
Maps to: Engineering Priority #5 · loop: Verification → Result.

- **Verification** step: confirm the result matches the intent (before/after diff).
- Transaction + rollback across multi-step plans; recovery on partial failure.
- Add a **second capability** (PDF/Word extraction) to prove the contract generalizes.

**Gate:** the full canonical loop runs reliably (~90%) across the supported task set,
with clean rollback on failure.

### M5 — UI Layer + polish (1 month)
Maps to: Engineering Priority #6 · chapter `07` product · UI direction in `CLAUDE.md`.

- Control Center shell; execution transparency (show understanding, plan, permission,
  steps, result); task history panel.
- Terminal-inspired dark design per the repo's UI direction.
- Error states and observability surfaced to the user.

**Gate:** a non-technical user can complete a real task unaided; the app is demoable.

### Buffer / stretch (only if closer to 6 months)
- Packaged installer (Windows/macOS), first-run onboarding.
- Put it in front of 10–50 real users.

**Stage 1 → Stage 2 gate:** repeat real usage + reliable execution. Traction here is what
later unlocks the Marketplace (Stage 2) and, only after real transaction volume exists,
the token layer (Stage 3).

---

## 3. Scope discipline

Deferred to later Stages (kept in the repo as north star, not built now): the full kernel
subsystem set, multi-model AI Router, mobile, cloud sync, marketplace, SDK, enterprise
features, and the token/crypto layer.

Decision rule for anything new: *"Is this required for the wedge to run and to get real
users?"* If no → defer.

---

## 4. If the deadline is nearer 3 months

Ship M0 → M1 → M2 → M3 only: a filesystem-only MVP that runs the canonical loop with a
permission gate. Defer M4 hardening and M5 polish to a fast-follow.

---

## 5. First code to write (M0)

Per the "define interfaces before implementation" rule, the very first artifacts are the
six interfaces in `schemas/`, followed by the minimal loop:
`Intent → LLM → ExecutionPlan → user approves → filesystem capability → event log → result`.

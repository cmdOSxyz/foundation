# Task

Version: 1.0
Status: Canonical
Category: Foundation

---

# 1. Purpose

This document defines the canonical meaning of **Task** within cmdOS.

A Task is the canonical unit of executable work within the platform.

Tasks transform Objectives into concrete execution activities that may be performed by Humans, Agents, Services, or automated systems.

Every governed Execution within cmdOS shall consist of one or more Tasks.

---

# 2. Definition

A **Task** is a governed unit of work that may be scheduled, assigned, executed, monitored, retried, completed, or cancelled.

A Task defines:

- work to perform
- execution target
- execution conditions
- required inputs
- expected outputs
- completion criteria

Tasks represent executable work.

They do not represent business intent.

---

# 3. Task Is Not an Objective

Objectives define desired outcomes.

Tasks define executable work.

```
Objective

↓

Task

↓

Execution
```

Multiple Tasks may satisfy a single Objective.

---

# 4. Task Is Not an Execution

Execution represents the complete operational process.

Tasks represent individual execution units.

```
Execution

├── Task A

├── Task B

└── Task C
```

Execution coordinates Tasks.

---

# 5. Task Is Not a Workflow

A Workflow defines execution structure.

Tasks are execution elements within that structure.

```
Workflow

↓

Task

↓

Execution
```

Tasks may exist independently of Workflows.

---

# 6. Task Characteristics

Every Task shall be:

- executable
- schedulable
- assignable
- observable
- traceable
- retryable
- governed

Tasks shall possess explicit execution boundaries.

---

# 7. Task Identity

Every Task shall possess a globally unique identifier.

Minimum metadata includes:

- Task ID
- Name
- Description
- Objective ID
- Status
- Priority
- Owner
- Assignee
- Version
- Creation Timestamp

Identity enables execution tracking.

---

# 8. Task Lifecycle

Every Task progresses through a managed lifecycle.

```
Created

↓

Scheduled

↓

Assigned

↓

Executing

↓

Completed

↓

Archived
```

Alternative terminal states include:

- Failed
- Cancelled
- Timed Out
- Skipped

Lifecycle transitions shall remain observable.

---

# 9. Task Inputs

Tasks may consume:

- Context
- Knowledge
- Memory
- Documents
- Resources
- Parameters
- Policies
- Previous Task outputs

Inputs shall remain governed.

---

# 10. Task Outputs

Tasks may produce:

- data
- artifacts
- events
- execution results
- status updates
- resource modifications
- subsequent Tasks

Outputs become governed execution artifacts.

---

# 11. Task Assignment

Tasks may be assigned to:

- Users
- Agents
- Organizations
- Services
- Connectors
- External Systems

Assignment authority shall remain explicit.

---

# 12. Task Scheduling

Tasks may execute according to scheduling policies.

Scheduling may consider:

- priority
- dependencies
- deadlines
- resource availability
- organizational policy
- execution windows

Scheduling remains implementation independent.

---

# 13. Task Dependencies

Tasks may depend upon other Tasks.

```
Task A

↓

Task B

↓

Task C
```

Dependencies shall remain explicit.

Circular dependencies shall be rejected.

---

# 14. Task Observability

Every significant Task event generates telemetry.

Examples include:

- created
- scheduled
- assigned
- started
- completed
- failed
- cancelled
- retried

Task history shall remain permanently observable.

---

# 15. Task Governance

Tasks remain governed throughout execution.

Governance includes:

- permissions
- policy validation
- auditing
- compliance
- approval requirements
- accountability

Governance establishes trustworthy execution.

---

# 16. Task Retry

Failed Tasks may be retried.

Retry policies may consider:

- retry limits
- delay strategies
- failure categories
- dependency state
- execution policy

Retries shall remain observable.

Every retry creates a new execution attempt.

---

# 17. Task Composition

Complex Tasks may contain child Tasks.

```
Parent Task

├── Child Task A

├── Child Task B

└── Child Task C
```

Parent Tasks remain responsible for overall completion.

---

# 18. Design Principles

Tasks follow these principles.

### Executable

Tasks define actionable work.

---

### Independent

Tasks maintain explicit execution boundaries.

---

### Governed

Tasks follow platform policies.

---

### Observable

Task execution generates telemetry.

---

### Retryable

Failures support governed recovery.

---

# 19. Task Invariants

The following statements shall always remain true.

1. Every Task has a unique identity.
2. Every Task belongs to one or more Objectives.
3. Every Task defines explicit completion criteria.
4. Task execution remains governed.
5. Task history remains observable.
6. Dependencies remain explicit.
7. Every retry is recorded.
8. Tasks never bypass platform governance.

These invariants apply throughout the cmdOS platform.

---

# 20. Relationship to the Specification

Task is the canonical executable work primitive within cmdOS.

Future specifications—including Task Engine, Workflow Engine, Execution Scheduler, Queue Framework, Retry Framework, Multi-Agent Coordination, Resource Scheduler, Autonomous Execution Framework, and Kernel—shall define behavior according to the canonical Task model established in this document.

Every governed unit of executable work within cmdOS shall be represented as one or more Tasks.

---

# 21. Summary

A Task is the governed, executable unit of work within cmdOS.

It transforms operational Objectives into concrete execution activities while preserving governance, observability, accountability, retryability, and execution isolation.

By separating Tasks from Objectives, Workflows, and Executions, cmdOS establishes a scalable execution architecture capable of supporting autonomous agents, distributed systems, enterprise orchestration, and reliable intelligent automation.

Task is therefore the canonical executable work primitive of the AI Execution Operating System.

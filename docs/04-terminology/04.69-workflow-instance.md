# Workflow Instance

Version: 1.0  
Status: Canonical  
Category: Foundation

---

# 1. Purpose

This document defines the canonical meaning of **Workflow Instance** within cmdOS.

Workflow Instance is the canonical runtime representation of a Workflow that has been activated and is currently being executed.

While a Workflow defines a reusable process structure, a Workflow Instance represents one specific occurrence of that process running under a particular Context, Actor, State, and Execution Environment.

Every running Workflow within cmdOS shall exist as one or more Workflow Instances.

---

# 2. Definition

A **Workflow Instance** is a runtime object created from a Workflow Definition that represents a single execution lifecycle of that Workflow.

A Workflow Instance defines:

- execution identity
- workflow state
- active Tasks
- completed Tasks
- execution Context
- participating Agents
- generated Results
- lifecycle status

Workflow Instance represents a Workflow in motion.

---

# 3. Workflow Instance Is Not a Workflow

Workflow defines the reusable blueprint.

Workflow Instance represents one runtime occurrence.

```
Workflow Definition

↓

Workflow Instance

↓

Execution Result
```

One Workflow may create unlimited Workflow Instances.

---

# 4. Workflow Instance Is Not Execution

Workflow Instance manages the complete process lifecycle.

Execution represents individual runtime operations inside that process.

```
Workflow Instance

↓

Task Instances

↓

Executions

↓

Results
```

A Workflow Instance may contain many Executions.

---

# 5. Workflow Instance Is Not Task

Task represents a defined unit of work.

Task Instance represents a specific runtime occurrence of that Task.

```
Task Definition

↓

Task Instance

↓

Action Execution
```

Workflow Instance coordinates Task Instances.

---

# 6. Workflow Instance Characteristics

Every Workflow Instance shall be:

- identifiable
- observable
- stateful
- traceable
- recoverable

Every Workflow Instance shall possess a globally unique identifier.

Minimum metadata includes:

- Workflow Instance ID
- Parent Workflow ID
- Trigger Source
- Owner
- Runtime Environment
- Creation Timestamp
- Current Status
- Version

Identity enables runtime tracking.

# 26. Workflow Instance Invariants

The following statements shall always remain true.

1. Every Workflow Instance has a unique identity.
2. Every Instance originates from a Workflow.
3. Every Instance has runtime State.
4. Every Instance contains controlled Tasks.
5. Every Instance execution is observable.
6. Every Instance follows Policies.
7. Every Instance preserves history.
8. Workflow Instances cannot bypass Governance.

These invariants apply throughout the cmdOS platform.

---

# 27. Relationship to the Specification

Workflow Instance is the canonical runtime process primitive within cmdOS.

Future specifications—including Workflow Engine, Automation Runtime, Scheduler, Task Manager, Execution Engine, Recovery System, and Agent Orchestration Framework—shall define behavior according to the canonical Workflow Instance model established in this document.

Every active Workflow within cmdOS shall exist as one or more Workflow Instances.

---

# 28. Summary

Workflow Instance transforms a static Workflow Definition into a living execution process.

It connects Triggers, Workflows, Tasks, Actions, Executions, Results, and Agents while maintaining state, governance, recovery, and observability.

By separating Workflow Instances from Workflow Definitions and Executions, cmdOS establishes a scalable runtime orchestration model capable of supporting autonomous Agents, enterprise automation, and distributed AI execution.

Workflow Instance is therefore the canonical runtime workflow primitive of the AI Execution Operating System.

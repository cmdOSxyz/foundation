# Context

Version: 1.0  
Status: Canonical  
Category: Foundation

---

# 1. Purpose

This document defines the canonical meaning of **Context** within cmdOS.

Context is the canonical representation of the information environment surrounding an Intent, Reasoning process, Inference operation, or Execution.

Context enables cmdOS to understand situations, interpret meaning, select appropriate actions, and produce relevant outcomes.

Every intelligent operation within cmdOS shall execute within an explicit Context.

---

# 2. Definition

A **Context** is a governed collection of information that describes the circumstances under which an operation occurs.

Context may include:

- current objective
- Actor information
- Workspace
- Project
- Session
- Memory references
- Knowledge references
- Current State
- Policies
- Constraints
- Environmental information

Context defines the situation.

It does not define permanent knowledge.

---

# 3. Context Is Not Memory

Memory stores information across time.

Context contains information relevant to the current operation.

```
Memory

↓

Retrieved Information

↓

Context

↓

Execution
```

Memory provides history.

Context provides relevance.

---

# 4. Context Is Not Knowledge

Knowledge represents reusable understanding.

Context represents information applied to a specific situation.

```
Knowledge

↓

Selected For

↓

Context

↓

Reasoning
```

Knowledge is general.

Context is situational.

---

# 5. Context Is Not State

State represents the condition of an Object.

Context represents the environment surrounding an operation.

```
State

+

Knowledge

+

Memory

+

Policies

↓

Context
```

State may become part of Context.

---

# 6. Context Characteristics

Every Context shall be:

- relevant
- bounded
- observable
- governed
- traceable
- temporary or persistent according to policy
- composable

Context shall contain only information required for its purpose.

---

# 7. Context Identity

Every Context instance shall possess a unique identifier.

Minimum metadata includes:

- Context ID
- Source Operation
- Owner
- Scope
- Version
- Creation Timestamp
- Expiration Policy

Identity enables traceability.

---

# 8. Context Lifecycle

Every Context progresses through a managed lifecycle.

```
Created

↓

Collected

↓

Assembled

↓

Used

↓

Updated

↓

Expired
```

Alternative states include:

- Archived
- Invalidated
- Destroyed

Lifecycle transitions shall remain observable.

---

# 9. Context Sources

Context may originate from:

- Intent
- Actor Identity
- Workspace
- Project
- Memory
- Knowledge
- Documents
- Events
- State
- External Systems

Every Context source shall remain traceable.

---

# 10. Context Assembly

Context Assembly combines relevant information for an operation.

Assembly may include:

- retrieval
- filtering
- ranking
- compression
- summarization
- transformation

Context Assembly shall preserve semantic meaning.

---

# 11. Context Types

cmdOS recognizes multiple Context categories.

---

## Execution Context

Information required during Execution.

Examples:

- Tasks
- Resources
- Permissions
- Runtime State

---

## Agent Context

Information used by Agents.

Examples:

- identity
- objectives
- memory
- capabilities

---

## Session Context

Information maintained during interaction.

Examples:

- conversation history
- temporary preferences
- active requests

---

## Project Context

Information related to a Project.

Examples:

- documents
- workflows
- resources
- objectives

---

## Environmental Context

External conditions affecting execution.

Examples:

- time
- location
- availability
- external events

---

# 12. Context Scope

Context shall define its scope.

Scope may include:

- Session
- Task
- Execution
- Project
- Workspace
- Organization

Information shall not exceed its authorized scope.

---

# 13. Context Expiration

Context may have expiration rules.

Expiration may depend upon:

- sensitivity
- relevance
- execution lifecycle
- organizational policy
- storage requirements

Expired Context shall not be used without validation.

---

# 14. Context Security

Context may contain sensitive operational information.

Security controls include:

- access control
- encryption
- isolation
- permission validation
- retention policies

Context access shall remain governed.

---

# 15. Context Observability

Every significant Context operation generates telemetry.

Examples include:

- created
- assembled
- accessed
- modified
- expired
- destroyed

Context history shall remain observable according to governance policies.

---

# 16. Context Management

Context management includes:

- creation
- retrieval
- prioritization
- compression
- updating
- removal

Context management shall optimize relevance and efficiency.

---

# 17. Context Window

A Context Window defines the amount of information available to an operation.

Context Window management may include:

- selection
- prioritization
- summarization
- truncation
- expansion

Context Window optimization shall preserve required meaning.

---

# 18. Dynamic Context

Context may change during execution.

Changes may occur due to:

- new Events
- State changes
- new Knowledge
- updated Policies
- Execution results

Dynamic Context updates shall remain observable.

---

# 19. Design Principles

Context follows these principles.

### Relevant

Context contains information useful for current operations.

---

### Controlled

Context access follows governance.

---

### Dynamic

Context adapts to changing situations.

---

### Traceable

Context sources remain identifiable.

---

### Efficient

Context avoids unnecessary information.

---

# 20. Context Invariants

The following statements shall always remain true.

1. Every intelligent operation has explicit Context.
2. Context sources are traceable.
3. Context remains governed.
4. Context scope is defined.
5. Context does not replace Memory.
6. Context does not replace Knowledge.
7. Context changes remain observable.
8. Context cannot bypass security policies.

These invariants apply throughout the cmdOS platform.

---

# 21. Relationship to the Specification

Context is the canonical situational information layer within cmdOS.

Future specifications—including Context Engine, Prompt Assembly System, Agent Runtime, Session Management, Retrieval Pipeline, Personalization System, Memory Engine, Knowledge Framework, and Kernel—shall define behavior according to the canonical Context model established in this document.

Every Reasoning, Inference, and Execution process within cmdOS shall operate through a Context.

---

# 22. Summary

Context is the information environment that allows cmdOS to understand and operate within a specific situation.

It connects Memory, Knowledge, State, Identity, Policies, and Objectives into a usable operational view for intelligent execution.

By separating Context from Memory, Knowledge, and State, cmdOS establishes a dynamic and controlled foundation for AI reasoning, personalization, autonomous agents, and execution intelligence.

Context is therefore the canonical situational intelligence primitive of the AI Execution Operating System.

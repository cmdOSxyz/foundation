# Action

Version: 1.0  
Status: Canonical  
Category: Foundation

---

# 1. Purpose

This document defines the canonical meaning of **Action** within cmdOS.

Action is the canonical representation of an atomic operation performed by an Actor, Agent, Tool, or System during an Execution.

Actions represent the smallest meaningful executable behavior within the AI Execution Operating System.

Every Execution within cmdOS shall consist of one or more Actions.

---

# 2. Definition

An **Action** is a governed atomic operation that changes State, produces a Result, creates an Artifact, or interacts with a Resource.

An Action defines:

- operation type
- execution target
- required Inputs
- expected Outputs
- required Tool
- execution conditions
- completion criteria

Actions represent what is being performed.

---

# 3. Action Is Not a Task

Task represents a unit of work.

Action represents an individual operation inside that Task.

```
Task

↓

Action

↓

Execution
```

A Task may contain multiple Actions.

---

# 4. Action Is Not a Tool

Tool provides the capability to perform operations.

Action represents one specific invocation of that capability.

```
Capability

↓

Tool

↓

Action

↓

Result
```

A Tool may execute many Actions.

---

# 5. Action Is Not an Execution

Action represents the operation definition.

Execution represents the runtime process performing that operation.

```
Action

↓

Execution Instance

↓

Result
```

An Action may have multiple Execution attempts.

---

# 6. Action Characteristics

Every Action shall be:

- atomic
- identifiable
- executable
- observable
- traceable
- governed
- reversible where possible

Actions shall have clearly defined behavior.

---

# 7. Action Identity

Every Action shall possess a globally unique identifier.

Minimum metadata includes:

- Action ID
- Parent Task
- Required Tool
- Actor
- Inputs
- Outputs
- Status
- Timestamp
- Version

Identity enables execution tracking.

---

# 8. Action Lifecycle

Every Action progresses through a managed lifecycle.

```
Created

↓

Validated

↓

Authorized

↓

Executing

↓

Completed

↓

Verified

↓

Archived
```

Alternative states include:

- Failed
- Rejected
- Cancelled
- Rolled Back
- Waiting Approval

Lifecycle transitions shall remain observable.

---

# 9. Action Sources

Actions may originate from:

- Planning Engine
- Workflow Engine
- Agent Decision
- Human Request
- System Event
- External Trigger

Every Action shall identify its source.

Examples:

- read data
- transform data
- analyze data
- store information

---

## Communication Action

Operations involving communication.

Examples:

- send message
- send email
- create notification

---

## System Action

Operations involving system control.

Examples:

- open application
- execute command
- modify file

---

## AI Action

Operations involving intelligence processing.

Examples:

- invoke model
- generate content
- classify information

---

## Blockchain Action

Operations involving decentralized systems.

Examples:

- query blockchain
- sign transaction
- execute contract

---

## Human Action

Operations requiring human participation.

Examples:

- approval
- review
- confirmation

---

# 11. Action Inputs

Actions may require:

- Parameters
- Context
- Resources
- Credentials
- Artifacts
- Previous Results

Inputs shall be validated before execution.

---

# 12. Action Outputs

Actions may produce:

- Results
- Artifacts
- Events
- State Changes
- New Actions

Outputs shall preserve provenance.

---

# 13. Action Preconditions

Every Action may define conditions required before execution.

Examples:

- permission granted
- Resource available
- previous Task completed
- Policy approved

Actions shall not execute without valid preconditions.

---

# 14. Action Effects

Every Action defines expected effects.

Effects may include:

- State changes
- Artifact creation
- External communication
- Resource consumption

Effects shall remain observable.

---

# 15. Action Authorization

Before execution, Actions shall be evaluated against:

- Identity
- Permissions
- Policies
- Context
- Risk requirements

Unauthorized Actions shall be rejected.

---

# 16. Action Execution

Action execution follows:

```
Action Created

↓

Validation

↓

Authorization

↓

Tool Selection

↓

Resource Allocation

↓

Execution

↓

Result
```

Every Action remains governed throughout execution.

---

# 17. Action Failure Handling

Actions may fail.

Failure handling may include:

- retry
- alternative Tool
- rollback
- compensation Action
- escalation

Failures shall remain traceable.

---

# 18. Action Reversibility

Actions may define reversibility.

Types:

```
Reversible Action

↓

Can return previous State
```

```
Irreversible Action

↓

Requires additional approval
```

Irreversible operations require stronger governance.

---

# 19. Action Dependencies

Actions may depend on other Actions.

Example:

```
Authenticate User

↓

Access Data

↓

Process Data
```

Dependencies determine execution order.

---

# 20. Action Observability

Every significant Action event generates telemetry.

Examples:

- created
- validated
- authorized
- started
- completed
- failed
- reversed

Action history shall remain observable.

---

# 21. Action Governance

Actions remain governed throughout lifecycle.

Governance includes:

- authorization
- permissions
- approval requirements
- audit records
- execution limits

Actions cannot exceed assigned authority.

---

# 22. Action Composition

Multiple Actions may combine into higher-level operations.

Example:

```
Open Browser

+

Search Information

+

Analyze Result

+

Create Report
```

forms a larger Workflow.

# 24. Design Principles

Actions follow these principles.

### Atomic

Actions represent clear operations.

---

### Observable

Every Action can be tracked.

---

### Governed

Every Action follows Policies.

---

### Composable

Actions combine into complex workflows.

---

### Traceable

Every Action preserves origin.

---

# 25. Action Invariants

The following statements shall always remain true.

1. Every Action has a unique identity.
2. Every Action belongs to a Task or Execution.
3. Every Action has defined Inputs and Outputs.
4. Every Action requires authorization.
5. Every Action execution is observable.
6. Every Action effect is traceable.
7. Actions cannot bypass Tools or Policies.
8. Action history is preserved.

These invariants apply throughout the cmdOS platform.

---

# 26. Relationship to the Specification

Action is the canonical atomic operation primitive within cmdOS.

Future specifications—including Action Engine, Function Calling System, Tool Runtime, Transaction System, Event Processing Layer, and Execution Engine—shall define behavior according to the canonical Action model established in this document.

Every operation performed by cmdOS shall be represented through one or more Actions.

---

# 27. Summary

Action represents the smallest executable behavior inside cmdOS.

It connects Tasks, Tools, Executions, Results, and State Changes while maintaining security, traceability, and governance.

By separating Actions from Tasks, Tools, and Executions, cmdOS establishes a precise execution model capable of supporting autonomous Agents, automation systems, desktop control, blockchain operations, and AI-native workflows.

Action is therefore the canonical atomic execution primitive of the AI Execution Operating System.

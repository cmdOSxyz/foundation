# Action

Version: 1.0
Status: Canonical
Category: Foundation

---

# 1. Purpose

This document defines the canonical meaning of **Action** within cmdOS.

An Action is the canonical atomic execution primitive within the platform.

Actions represent the smallest governed unit of work that may be executed, validated, audited, retried, compensated, or rolled back.

Every Task executed within cmdOS shall consist of one or more Actions.

---

# 2. Definition

An **Action** is a governed atomic operation performed by an execution target to produce a specific and measurable effect.

An Action defines:

- operation
- execution target
- required inputs
- expected outputs
- execution conditions
- completion criteria

Actions represent atomic execution.

They do not represent workflows or business objectives.

---

# 3. Action Is Not a Task

Tasks represent executable work.

Actions represent atomic operations within that work.

```
Task

├── Action A

├── Action B

└── Action C
```

A Task may contain one or many Actions.

---

# 4. Action Is Not an Execution

Execution coordinates the complete runtime process.

Actions perform individual atomic operations.

```
Execution

↓

Task

↓

Action
```

Execution orchestrates.

Actions execute.

---

# 5. Action Is Not a Capability

Capabilities describe what the platform can perform.

Actions represent a specific invocation of that capability.

```
Capability

↓

Action
```

A Capability may support countless Actions.

---

# 6. Action Characteristics

Every Action shall be:

- atomic
- observable
- traceable
- governed
- deterministic when configured
- retryable
- independently auditable

Actions shall possess explicit execution boundaries.

---

# 7. Action Identity

Every Action shall possess a globally unique identifier.

Minimum metadata includes:

- Action ID
- Task ID
- Execution ID
- Capability ID
- Status
- Version
- Executor
- Creation Timestamp

Identity enables execution tracking and auditing.

---

# 8. Action Lifecycle

Every Action progresses through a managed lifecycle.

```
Created

↓

Validated

↓

Executing

↓

Completed

↓

Archived
```

Alternative terminal states include:

- Failed
- Cancelled
- Timed Out
- Rolled Back
- Compensated

Lifecycle transitions shall remain observable.

---

# 9. Action Inputs

Actions may consume:

- parameters
- Context
- Knowledge
- Memory
- Resources
- Documents
- previous Action outputs
- execution policies

Inputs shall remain immutable during Action execution unless explicitly permitted.

---

# 10. Action Outputs

Actions may produce:

- execution results
- Artifacts
- Events
- resource updates
- status changes
- audit records
- subsequent Actions

Outputs become governed execution artifacts.

---

# 11. Action Types

Actions may belong to multiple categories.

Examples include:

- AI inference
- tool invocation
- API request
- database operation
- filesystem operation
- connector operation
- approval action
- notification action

Additional Action categories may be introduced without changing the canonical model.

---

# 12. Action Execution

An Action shall execute according to its execution policy.

Execution may require:

- permissions
- resource allocation
- dependency validation
- policy evaluation
- execution authorization

Execution shall remain deterministic whenever configured.

---

# 13. Action Atomicity

An Action shall be treated as an atomic execution unit.

An Action shall either:

- complete successfully

or

- terminate according to defined failure policies.

Partial completion shall not be externally observable unless explicitly supported by future specifications.

---

# 14. Action Observability

Every significant Action event generates telemetry.

Examples include:

- created
- validated
- executing
- completed
- failed
- cancelled
- retried
- rolled back

Action history shall remain permanently observable.

---

# 15. Action Governance

Actions remain governed throughout execution.

Governance includes:

- permission validation
- policy enforcement
- compliance
- auditing
- execution authorization
- accountability

Governance establishes trustworthy execution.

---

# 16. Action Recovery

Failed Actions may support recovery mechanisms.

Recovery strategies may include:

- retry
- rollback
- compensation
- escalation
- manual intervention

Recovery behavior shall remain governed and observable.

---

# 17. Action Composition

Complex operations may consist of multiple Actions.

```
Operation

├── Action A

├── Action B

├── Action C

└── Action D
```

Each Action remains independently observable and auditable.

---

# 18. Design Principles

Actions follow these principles.

### Atomic

Actions represent indivisible execution units.

---

### Governed

Actions follow platform policies.

---

### Observable

Every Action generates execution telemetry.

---

### Recoverable

Failures support governed recovery strategies.

---

### Traceable

Every Action preserves execution history.

---

# 19. Action Invariants

The following statements shall always remain true.

1. Every Action has a unique identity.
2. Every Action belongs to exactly one Task.
3. Every Action has explicit execution boundaries.
4. Action execution remains governed.
5. Every Action is observable.
6. Every Action is independently auditable.
7. Recovery activities remain traceable.
8. Actions never bypass platform governance.

These invariants apply throughout the cmdOS platform.

---

# 20. Relationship to the Specification

Action is the canonical atomic execution primitive within cmdOS.

Future specifications—including Execution Engine, Action Registry, Permission Engine, Rollback Framework, Transaction Engine, Audit Trail, Retry Framework, Workflow Runtime, and Kernel—shall define behavior according to the canonical Action model established in this document.

Every atomic execution operation within cmdOS shall be represented as one or more Actions.

---

# 21. Summary

An Action is the governed atomic execution primitive within cmdOS.

It transforms executable work into concrete operations while preserving atomicity, governance, observability, accountability, and recoverability.

By separating Actions from Tasks, Capabilities, and Executions, cmdOS establishes a reliable execution architecture capable of supporting transactional workflows, distributed systems, autonomous agents, and enterprise-scale intelligent automation.

Action is therefore the canonical atomic execution primitive of the AI Execution Operating System.

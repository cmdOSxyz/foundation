# Task

Version: 1.0  
Status: Canonical  
Category: Foundation

---

# 1. Purpose

This document defines the canonical meaning of **Task** within cmdOS.

Task is the canonical representation of an individual unit of work required to achieve an Objective through a Workflow and Execution process.

Tasks transform Plans into manageable operational units.

Every operational process within cmdOS shall be composed of one or more Tasks.

---

# 2. Definition

A **Task** is a governed unit of work that defines a specific activity required to produce a desired intermediate or final outcome.

A Task defines:

- required operation
- expected outcome
- inputs
- dependencies
- assigned Actor
- required Resources
- completion criteria

Tasks represent work that must be performed.

---

# 3. Task Is Not an Objective

Objective defines the measurable target.

Task defines the work required to achieve that target.

```
Objective

↓

Task

↓

Result
```

Objectives describe success.

Tasks describe activities.

---

# 4. Task Is Not an Action

Task represents a unit of work.

Action represents the atomic operation performed inside a Task.

```
Task

↓

Action

↓

Execution
```

A Task may contain multiple Actions.

---

# 5. Task Is Not an Execution

Task defines what work should occur.

Execution represents the runtime attempt to perform that work.

```
Task

↓

Execution

↓

Result
```

A Task may have multiple Execution attempts.

---

# 6. Task Characteristics

Every Task shall be:

- identifiable
- actionable
- measurable
- assignable
- observable
- governed
- traceable

Tasks shall have clear completion requirements.

---

# 7. Task Identity

Every Task shall possess a globally unique identifier.

Minimum metadata includes:

- Task ID
- Parent Objective
- Parent Workflow
- Owner
- Assigned Actor
- Status
- Priority
- Creation Timestamp
- Version

Identity enables task management.

---

# 8. Task Lifecycle

Every Task progresses through a managed lifecycle.

```
Created

↓

Assigned

↓

Ready

↓

Running

↓

Completed

↓

Verified

↓

Archived
```

Alternative states include:

- Blocked
- Failed
- Cancelled
- Waiting Approval
- Suspended

Lifecycle transitions shall remain observable.

---

# 9. Task Sources

Tasks may originate from:

- Planning Engine
- Workflow

- Agent Reasoning
- System Event
- External Trigger

Every Task shall identify its source.

- monitoring

---

Performed automatically by system components.

Examples:

- synchronization

- maintenance

Represents direct operational work.

- generating output

---

## Verification Task

- auditing
- quality checking

# 24. Design Principles

Tasks follow these principles.

### Actionable

Tasks represent executable work.

---

### Traceable

Every Task has an origin.

---

### Composable

Tasks combine into larger processes.

---

### Observable

Task progress remains visible.

---

### Governed

Task execution follows policies.

---

# 25. Task Invariants

The following statements shall always remain true.

1. Every Task belongs to a Workflow or Objective.
2. Every Task has an owner or assigned Actor.
3. Every Task has completion criteria.
4. Every Task lifecycle transition is observable.
5. Every Task execution follows Policies.
6. Every Task output is traceable.
7. Tasks cannot bypass authorization.
8. Task history is preserved.

These invariants apply throughout the cmdOS platform.

---

# 26. Relationship to the Specification

Task is the canonical unit-of-work primitive within cmdOS.

Future specifications—including Task Manager, Scheduler, Queue System, Agent Task System, Workflow Engine, Execution Engine, and Human Collaboration Framework—shall define behavior according to the canonical Task model established in this document.

Every operational workflow within cmdOS shall consist of one or more Tasks.

---

# 27. Summary

Task represents the smallest meaningful unit of work that contributes toward achieving an Objective.

It connects Planning, Workflow, Actions, Tools, and Execution while maintaining ownership, observability, and governance.

By separating Tasks from Objectives, Actions, and Executions, cmdOS creates a scalable work management foundation for autonomous Agents, human collaboration, automation systems, and AI-driven operating environments.

Task is therefore the canonical unit-of-work primitive of the AI Execution Operating System.

# Context

Version: 1.0
Status: Canonical
Category: Foundation

---

# 1. Purpose

This document defines the canonical meaning of **Context** within cmdOS.

Intent describes **what** should be accomplished.

Context describes **the conditions under which that objective should be accomplished**.

Every trusted execution is derived from the combination of Intent and Context.

---

# 2. Definition

A **Context** is the complete collection of information required to correctly interpret and execute an Intent.

Context describes the execution environment.

It provides meaning.

It does not define objectives.

---

# 3. Intent vs Context

Intent answers:

> What should happen?

Context answers:

> Under what conditions should it happen?

Example:

Intent

```
Summarize today's meetings.
```

Context

- today's calendar
- meeting transcripts
- preferred language
- user identity
- available AI models
- company policies

Together they define the execution request.

---

# 4. Context Is Not Intent

Context shall never modify the user's objective.

Instead:

```
Intent

+

Context

↓

Planning
```

Context influences execution.

Intent defines purpose.

The two concepts remain independent.

---

# 5. Context Categories

Context may originate from multiple sources.

Examples include:

### Identity Context

- user identity
- organization
- roles
- delegated authority

---

### Environment Context

- operating system
- device
- runtime
- network
- location

---

### Temporal Context

- current time
- time zone
- deadlines
- schedules

---

### Knowledge Context

- documents
- memory
- historical interactions
- retrieved information

---

### Security Context

- permissions
- trust level
- authentication status
- policy scope

---

### Execution Context

- workflow state
- active session
- execution history
- resource availability

Each category contributes to accurate execution.

---

# 6. Context Lifecycle

Context continuously evolves throughout execution.

```
Collected

↓

Validated

↓

Resolved

↓

Merged

↓

Consumed

↓

Archived
```

Context may be refreshed during long-running workflows.

---

# 7. Context Is Dynamic

Unlike Intent, Context may legitimately change.

Examples:

- current time
- battery level
- available memory
- network status
- AI provider availability
- system load

Dynamic Context enables adaptive execution.

---

# 8. Context Is Explicit

Every Context used during execution shall be identifiable.

Hidden contextual assumptions are prohibited.

Execution records should identify:

- Context sources
- Context version
- collection time
- validation status

Explicit Context improves reproducibility.

---

# 9. Context Resolution

Information often arrives from multiple providers.

Example:

```
Calendar

+

Memory

+

Filesystem

+

Cloud

+

Policy

↓

Unified Context
```

The Context Engine resolves conflicts before planning begins.

---

# 10. Context and Planning

Planning consumes both Intent and Context.

```
Intent

+

Context

↓

Planning Engine

↓

Execution Plan
```

Planning without Context is incomplete.

Context without Intent has no objective.

---

# 11. Context and Policy

Policy evaluation depends upon Context.

Examples include:

- user role
- organization
- location
- security level
- execution environment

Policy decisions shall never rely upon hidden contextual information.

---

# 12. Context and AI

AI systems require Context to produce reliable recommendations.

Examples include:

- previous conversation
- retrieved knowledge
- available tools
- user preferences
- execution history

Context improves relevance while preserving the original Intent.

---

# 13. Context Isolation

Context shall remain isolated between independent executions.

Execution A shall never unintentionally inherit Context from Execution B.

Isolation protects:

- privacy
- correctness
- security
- determinism

Context leakage is prohibited.

---

# 14. Context and Observability

Every Context shall be observable.

Observable metadata includes:

- origin
- timestamp
- version
- validation status
- consuming execution
- lifecycle events

Observability supports auditing and debugging.

---

# 15. Design Principles

Context follows these principles.

### Explicit

Every Context source is identifiable.

---

### Independent

Context never replaces Intent.

---

### Dynamic

Context evolves with execution.

---

### Observable

Context changes generate telemetry.

---

### Governed

Context remains subject to security and policy.

---

# 16. Context Invariants

The following statements shall always remain true.

1. Context complements Intent.
2. Context never changes Intent.
3. Context may evolve during execution.
4. Context sources remain explicit.
5. Context is isolated between executions.
6. Context is validated before planning.
7. Context participates in policy evaluation.
8. Context is observable throughout its lifecycle.

These invariants apply throughout the cmdOS platform.

---

# 17. Relationship to the Specification

Context is one of the foundational objects of cmdOS.

Future specifications—including Context Engine, Planning Engine, Agent Runtime, Policy Engine, Memory Framework, Knowledge System, Execution Runtime, and Kernel—shall consume canonical Context objects.

Context shall always be processed independently of Intent while remaining tightly associated throughout execution.

---

# 18. Summary

Intent defines the destination.

Context defines the conditions for reaching it.

By separating objectives from environmental information, cmdOS enables adaptive, secure, deterministic, and trustworthy execution across changing environments.

Together, Intent and Context form the canonical input to every execution within the AI Execution Operating System.

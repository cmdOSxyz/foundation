# cmdOS Context
# Part 10.35.22 — Permission Manager

Version: 1.0
Status: Stable
Category: Kernel Security

---

# 1. Purpose

This document defines the Permission Manager of cmdOS.

The Permission Manager is the canonical authorization component responsible for determining which actions an authenticated identity is permitted to perform.

It answers one question:

**"What is this identity allowed to do?"**

Permission evaluation occurs after identity verification and before execution.

---

# 2. Philosophy

Authentication identifies.

Authorization permits.

Permissions are explicit.

Permissions are least-privilege.

Permissions are never assumed.

Every permission is auditable.

---

# 3. Canonical Principle

Every permission evaluation follows

Identity Resolution

↓

Permission Discovery

↓

Scope Resolution

↓

Policy Evaluation

↓

Capability Verification

↓

Decision

↓

Execution

↓

Audit

Permission validation always precedes execution.

---

# 4. Responsibilities

The Permission Manager is responsible for

- Permission assignment
- Permission evaluation
- Scope resolution
- Role mapping
- Attribute evaluation
- Capability authorization
- Delegation management
- Permission auditing
- Permission caching
- Authorization events

The Permission Manager does not

- Authenticate identities
- Execute commands
- Store secrets
- Interpret business logic

---

# 5. Design Principles

- Least Privilege
- Explicit Authorization
- Fine-Grained Control
- Namespace Awareness
- Deterministic Evaluation
- Observable
- Auditable
- Provider Independent

---

# 6. Permission Object

Every permission contains

- Permission ID
- Permission Name
- Permission Type
- Namespace
- Scope
- Resource
- Allowed Actions
- Constraints
- Status
- Metadata

Permission identifiers are immutable.

---

# 7. Permission Models

The Permission Manager supports

- Role-Based Access Control (RBAC)
- Attribute-Based Access Control (ABAC)
- Capability-Based Authorization
- Resource-Based Authorization
- Context-Aware Authorization
- Time-Based Authorization

Multiple models may be evaluated together.

---

# 8. Permission Scope

Permissions may apply to

- System
- Organization
- Workspace
- Project
- Namespace
- Resource
- Capability
- Object
- Session

Permission scope is explicitly defined.

---

# 9. Role Management

Roles group related permissions.

Examples include

- Administrator
- Operator
- Developer
- Auditor
- AI Agent
- Read Only
- Guest
- Service Account

Roles simplify permission assignment without replacing policy evaluation.

---

# 10. Delegation

Permissions may be delegated through

- Temporary delegation
- Permanent delegation
- Conditional delegation
- Scoped delegation

Delegation is fully auditable.

---

# 11. Capability Authorization

Every capability invocation requires

- Identity verification
- Permission validation
- Policy approval
- Namespace verification

Unauthorized capability execution is prohibited.

---

# 12. Context-Aware Authorization

Permission evaluation considers

- Namespace
- Device
- Session
- Time
- Network
- Risk Level
- Runtime Context

Authorization decisions adapt to execution context.

---

# 13. Events

The Permission Manager emits

- PermissionGranted
- PermissionDenied
- PermissionAssigned
- PermissionRevoked
- RoleAssigned
- DelegationCreated
- AuthorizationCompleted

Events are immutable.

---

# 14. Security

Security includes

- Least privilege
- Permission inheritance control
- Delegation validation
- Namespace isolation
- Audit logging
- Continuous authorization

Permissions are continuously enforced throughout execution.

---

# 15. Observability

Metrics include

- Authorization requests
- Permission grants
- Permission denials
- Delegation count
- Role assignments
- Evaluation latency
- Authorization failures

Sensitive authorization data is never exposed.

---

# 16. Failure Modes

Possible failures include

- Missing permission
- Invalid role
- Namespace mismatch
- Delegation expired
- Policy conflict
- Authorization timeout
- Capability restriction

Failures default to deny.

The Permission Manager integrates with

- Identity Manager

---

# 19. Trust Boundary

Permissions never establish identity.

Permissions only authorize verified identities.

Every authorization decision requires successful authentication and policy evaluation.

---

# 20. Architectural Invariants

1. Every authorization references a verified identity.
2. Every permission has a defined scope.
3. Authorization follows authentication.
4. Least privilege is enforced.
5. Permission evaluation is deterministic.
6. Delegations are explicit and auditable.
7. Unauthorized execution is impossible.
8. Namespace boundaries are respected.
9. Permission changes generate immutable events.
10. Authorization decisions are fully auditable.

---

# 21. Summary

The Permission Manager is the canonical authorization system of cmdOS.

By combining fine-grained permissions, multiple authorization models, namespace-aware evaluation and deterministic policy enforcement, it ensures that every authenticated identity performs only explicitly permitted actions throughout the Kernel.

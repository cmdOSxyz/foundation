# cmdOS Context
# Part 10.35.03 — Kernel Command Manager

Version: 1.0
Status: Stable
Category: Kernel Infrastructure

---

# 1. Purpose

This document defines the Kernel Command Manager of cmdOS.

The Command Manager is responsible for transforming validated intent into deterministic executable commands.

It answers one question:

**"What should the system do?"**

---

# 2. Philosophy

Intent expresses desire.

Commands express action.

Commands are immutable execution requests.

Commands never execute themselves.

Execution belongs to the Execution Engine.

---

# 3. Canonical Principle

Every command follows:

Intent

↓

Validation

↓

Command Creation

↓

Policy Validation

↓

Registration

↓

Execution Planning

↓

Execution

↓

Completion

↓

Audit

No command may bypass this lifecycle.

---

# 4. Responsibilities

The Command Manager is responsible for:

- Command creation
- Command validation
- Command registration
- Command versioning
- Command metadata
- Command lifecycle
- Command discovery
- Command relationships
- Command audit

The Command Manager does not:

- Execute commands
- Modify canonical state
- Interpret natural language
- Manage transactions
- Schedule execution

---

# 5. Design Principles

- Immutable Commands
- Explicit Intent
- Version Awareness
- Policy Governed
- Observable
- Deterministic
- Auditable
- Recoverable

---

# 6. Command Object

Every command contains:

- Command ID
- Command Type
- Namespace
- Object Reference
- Requested Action
- Parameters
- Identity
- Policy Context
- Creation Time
- Version
- Metadata

Command IDs are immutable.

---

# 7. Command Types

Examples include:

- Create
- Read
- Update
- Delete
- Execute
- Invoke
- Schedule
- Cancel
- Pause
- Resume
- Rollback

Custom command types must follow kernel contracts.

---

# 8. Lifecycle

```text
Created
    ↓
Validated
    ↓
Registered
    ↓
Planned
    ↓
Executing
    ↓
Completed
```

Failure paths:

- Rejected
- Cancelled
- Failed
- Rolled Back

---

# 9. Validation

Validation includes:

- Schema
- Namespace
- Identity
- Object existence
- Capability
- Policy
- Parameters
- Version
- Dependencies

Invalid commands are rejected before planning.

---

# 10. Identity

Every command is associated with:

- Requester
- Namespace
- Session
- Transaction
- Execution Context

Anonymous commands are prohibited.

---

# 11. Parameters

Parameters must be:

- Typed
- Validated
- Serializable
- Version compatible

Hidden parameters are prohibited.

---

# 12. Relationships

Commands may reference:

- Objects
- Transactions
- Execution Plans
- Runtime Targets
- Capabilities
- Policies

Relationships remain immutable.

---

# 13. Planning Integration

Validated commands are forwarded to the Execution Planner.

The planner determines:

- Task graph
- Dependencies
- Scheduling strategy
- Runtime requirements

The Command Manager never decides execution order.

---

# 14. Events

Events include:

- CommandCreated
- CommandValidated
- CommandRejected
- CommandRegistered
- CommandPlanned
- CommandCompleted
- CommandFailed
- CommandCancelled

Events are immutable.

---

# 15. Security

Security requirements include:

- Identity validation
- Namespace isolation
- Policy enforcement
- Capability checks
- Audit logging
- Least privilege

---

# 16. Observability

Metrics include:

- Commands created
- Validation failures
- Planning latency
- Execution success rate
- Command failures
- Average lifecycle duration

Sensitive parameters must never appear in telemetry.

---

# 17. Failure Modes

Failure modes include:

- Invalid schema
- Missing object
- Policy denial
- Namespace violation
- Unsupported action
- Dependency failure
- Planner rejection

Failures prevent execution.

---

Intent

# 19. Architectural Integration

The Command Manager integrates with:

- Object Manager
- State Manager
- Execution Planner
- Transaction Manager
- Lock Manager
- Consistency Manager
- Event Bus
- Security
- Observability

---

# 20. Summary

The Kernel Command Manager transforms validated intent into deterministic executable commands.

It guarantees every action is explicit, validated, observable and compliant with kernel contracts before execution.

# cmdOS Context
# Part 10.35.17 — Snapshot Manager

Version: 1.0
Status: Stable
Category: Kernel Persistence

---

# 1. Purpose

This document defines the Snapshot Manager of cmdOS.

The Snapshot Manager creates, stores and manages verified snapshots of canonical system state to accelerate recovery, replay and synchronization.

It answers one question:

**"What is the latest trusted checkpoint of the system?"**

Snapshots reduce replay cost while preserving deterministic recovery.

---

# 2. Philosophy

Events are the complete history.

Snapshots are optimized checkpoints.

Snapshots never replace events.

Snapshots are disposable.

Events remain the permanent source of truth.

---

# 3. Canonical Principle

Every snapshot follows

State Verification

↓

Snapshot Request

↓

Consistency Validation

↓

Snapshot Creation

↓

Integrity Verification

↓

Persistence

↓

Publication

↓

Recovery Availability

Snapshots are created only from verified canonical state.

---

# 4. Responsibilities

The Snapshot Manager is responsible for

- Snapshot creation
- Snapshot storage
- Snapshot indexing
- Snapshot validation
- Snapshot restoration
- Snapshot pruning
- Snapshot versioning
- Snapshot integrity
- Recovery acceleration
- Snapshot auditing

The Snapshot Manager does not

- Modify events
- Execute commands
- Validate policies
- Replace Event Store

---

# 5. Design Principles

- Immutable Snapshots
- Verified State Only
- Fast Recovery
- Version Aware
- Observable
- Recoverable
- Auditable
- Provider Independent

---

# 6. Snapshot Object

Every snapshot contains

- Snapshot ID
- Snapshot Version
- Aggregate ID
- Namespace
- State Version
- Event Position
- Creation Timestamp
- Integrity Hash
- Compression Method
- Metadata

Snapshots are immutable.

---

# 7. Snapshot Triggers

Snapshots may be created after

- Fixed event count
- Time interval
- Transaction completion
- Manual request
- State size threshold
- Recovery checkpoint

Trigger policies are configurable.

---

# 8. Storage Model

Snapshots may be stored as

- Binary image
- Structured document
- Serialized objects
- Distributed storage
- Encrypted archive

Storage format does not affect logical behavior.

---

# 9. Restoration

Recovery using snapshots follows

Snapshot

↓

Integrity Check

↓

Restore State

↓

Replay Remaining Events

↓

Consistency Verification

↓

Canonical State

Replay begins after the snapshot event position.

---

# 10. Version Management

Snapshots include

- Snapshot Version
- Schema Version
- Kernel Version
- State Version

Version compatibility is validated before restoration.

---

# 11. Retention

Retention policies define

- Latest snapshot
- Rolling snapshots
- Daily snapshots
- Weekly snapshots
- Archived snapshots

Expired snapshots may be deleted without affecting historical correctness.

---

# 12. Integrity

Snapshot integrity uses

- Cryptographic hashes
- Digital signatures
- Checksums
- Version validation

Corrupted snapshots are rejected.

---

# 13. Events

The Snapshot Manager emits

- SnapshotCreated
- SnapshotValidated
- SnapshotStored
- SnapshotRestored
- SnapshotDeleted
- SnapshotExpired

Events are immutable.

---

# 14. Security

Security includes

- Encryption at rest
- Namespace isolation
- Access control
- Integrity verification
- Audit logging

Unauthorized snapshot access is prohibited.

---

# 15. Observability

Metrics include

- Snapshot count
- Snapshot size
- Creation latency
- Restore latency
- Integrity failures
- Snapshot age
- Compression ratio

Sensitive snapshot contents remain protected.

---

# 16. Failure Modes

Possible failures include

- Corrupted snapshot
- Version mismatch
- Missing snapshot
- Restore failure
- Integrity failure
- Storage failure

Recovery falls back to full event replay when necessary.

# 18. Architectural Integration

The Snapshot Manager integrates with

- Event Store
- Recovery Manager
- State Manager
- Transaction Manager
- Consistency Manager
- Observability

Snapshots optimize—but never replace—the Event Store.

---

# 19. Trust Boundary

Snapshots are trusted only after integrity verification.

Canonical truth remains derived from the Event Store plus validated snapshots.

---

# 20. Architectural Invariants

1. Snapshots originate only from verified canonical state.
2. Snapshots are immutable.
3. Every snapshot has a unique identifier.
4. Every snapshot references an Event Store position.
5. Integrity is verified before restoration.
6. Recovery always validates restored state.
7. Snapshot deletion never removes historical truth.
8. Version compatibility is mandatory.
9. Replay resumes after snapshot position.
10. Event Store remains authoritative.

---

# 21. Summary

The Snapshot Manager accelerates recovery by maintaining immutable checkpoints of verified canonical state.

It enables fast restoration, efficient replay and scalable event sourcing while preserving the Event Store as the permanent source of historical truth.

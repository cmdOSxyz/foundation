# cmdOS Context
# Part 10.35.02 — Kernel State Manager

Version: 1.0
Status: Stable
Category: Kernel Infrastructure

---

## 1. Purpose

This document defines the Kernel State Manager of cmdOS.

The State Manager is the canonical authority for current system state and all approved state transitions.

It answers one question:

**What state is this object in?**

---

## 2. Philosophy

State is explicit.

State is versioned.

State changes only through validated transitions.

Temporary memory is not canonical state.

Cache is not canonical state.

Observed runtime output becomes state only after verification and acceptance.

---

## 3. Canonical Principle

Every state transition follows:

State Read

↓

Version Validation

↓

Transition Request

↓

Policy Validation

↓

Precondition Check

↓

Transaction Boundary

↓

State Mutation

↓

Consistency Verification

↓

Version Increment

↓

Event Emission

↓

Persistence

No canonical state may be mutated outside this flow.

---

## 4. Responsibilities

The State Manager is responsible for:

- Canonical state ownership
- State creation
- State reads
- State transitions
- Transition validation
- State versioning
- State history references
- State snapshots
- State reconciliation
- Transition events
- State lifecycle enforcement

The State Manager does not:

- Interpret user intent
- Execute runtime actions
- Authorize identity
- Schedule tasks
- Manage temporary memory
- Serve as a cache

---

## 5. Design Principles

- Single Canonical Authority
- Explicit Transitions
- Versioned State
- Deterministic Mutation
- Transactional Changes
- Policy Governed
- Observable
- Recoverable
- No Hidden State
- No Direct AI Mutation

---

## 6. State Object

Every canonical state is represented by a State Object.

A State Object contains:

- State ID
- Object Reference
- Namespace
- State Type
- Current Value
- Previous Version
- Current Version
- Lifecycle State
- Creation Time
- Last Updated Time
- Transition Reference
- Transaction Reference
- Policy Reference
- Integrity Metadata
- Audit Metadata

---

## 7. State Types

Canonical state types may include:

- Object State
- Execution State
- Task State
- Transaction State
- Runtime State
- Capability State
- Resource State
- Provider State
- Recovery State
- Configuration State
- Security State

Each state type has an explicit schema and transition model.

---

## 8. State Lifecycle

A generic lifecycle may include:

```text
Uninitialized
    ↓
Initialized
    ↓
Active
    ↓
Suspended
    ↓
Completed
    ↓
Archived
```

Alternative lifecycles require an approved schema.

Lifecycle transitions must never be inferred silently.

---

## 9. Transition Model

A transition contains:

- Transition ID
- Source State
- Target State
- Object Reference
- Requested By
- Command Reference
- Transaction Reference
- Preconditions
- Policy Requirements
- Timestamp
- Result
- Failure Reason

Transitions are immutable records.

---

## 10. Preconditions

Before a transition is accepted, the State Manager validates:

- Current version
- Expected source state
- Object existence
- Namespace
- Ownership
- Policy
- Lock status
- Transaction status
- Dependency state
- Schema compatibility

A failed precondition prevents mutation.

---

## 11. Versioning

Each successful mutation increments the state version.

Versioning protects against:

- Lost updates
- Stale writes
- Race conditions
- Replay errors
- Incorrect recovery
- Conflicting transitions

Optimistic or pessimistic concurrency control may be selected by policy.

---

## 12. Read Operations

State reads may support:

- Read by State ID
- Read by Object ID
- Read by Namespace
- Read by Version
- Read Current State
- Read Historical Reference
- Read State Projection

Authorization applies to every read.

---

## 13. Write Operations

Writes must include:

- Object reference
- Expected version
- Requested transition
- Transaction context
- Identity context
- Policy context
- Audit context

Anonymous writes are prohibited.

---

## 14. Transactions

State mutation occurs within an explicit transaction boundary.

The Transaction Manager coordinates:

- Begin
- Prepare
- Commit
- Rollback
- Compensation

The State Manager records only accepted outcomes.

---

## 15. Consistency Integration

The Consistency Manager evaluates:

```text
Desired Outcome
      ↓
Observed State
      ↓
Policy
      ↓
Consistency Result
```

A runtime action does not automatically become canonical state.

The observed result must first satisfy consistency requirements.

---

## 16. Lock Integration

The Lock Manager protects concurrent state mutation.

Locks may apply to:

- Object
- State record
- Namespace
- Resource
- Transaction scope

State writes must respect active lock policy.

---

## 17. Event Integration

The State Manager emits immutable events including:

- StateCreated
- StateRead
- StateTransitionRequested
- StateTransitionAccepted
- StateTransitionRejected
- StateUpdated
- StateVersionCreated
- StateReconciled
- StateArchived
- StateRestored

Events must not expose protected data.

---

## 18. Persistence

Canonical state must be durable according to policy.

Persistence may use:

- State database
- Event sourcing
- Snapshot plus journal
- Transaction log
- Replicated storage

The selected mechanism must preserve version and integrity guarantees.

---

## 19. Snapshot Integration

Snapshots capture state at a known version.

A snapshot contains:

- Snapshot ID
- State version
- Object references
- Timestamp
- Integrity hash
- Schema version
- Recovery metadata

Snapshots support recovery and audit.

---

## 20. Recovery

Recovery may restore state using:

- Last committed version
- Event replay
- Checkpoint
- Snapshot
- Transaction journal
- Runtime reconciliation

Unverified runtime output must not be promoted to canonical state during recovery.

---

## 21. Reconciliation

Reconciliation compares:

- Expected state
- Persisted state
- Runtime-observed state
- Event history
- Transaction status

Reconciliation may result in:

- Confirmed
- Repaired
- Compensated
- Rolled Back
- Quarantined
- Escalated

---

## 22. State Projection

Projections are derived views of canonical state.

Projections may be:

- Cached
- Indexed
- Aggregated
- Materialized
- Rebuilt

A projection is never authoritative over its source state.

---

## 23. Distributed State

Distributed deployments require:

- Version coordination
- Conflict detection
- Replication policy
- Partition handling
- Leader or quorum rules
- Reconciliation
- Fencing
- Idempotency

No distributed node may silently overwrite a newer canonical version.

---

## 24. Security

Security requirements include:

- Namespace isolation
- Ownership validation
- Least privilege
- Policy enforcement
- Integrity checks
- Protected state encryption
- Auditable transitions
- Default deny
- No direct model access to mutation APIs

---

## 25. Observability

Metrics include:

- State reads
- State writes
- Transition success rate
- Transition rejection rate
- Version conflicts
- Reconciliation count
- Recovery count
- Average transition latency
- Persistence latency
- Lock wait time

Sensitive values must never appear in telemetry.

---

## 26. Failure Modes

Failure modes include:

- Invalid transition
- Version conflict
- Missing object
- Namespace violation
- Policy denial
- Lock timeout
- Persistence failure
- Transaction rollback
- Corrupted state
- Schema mismatch
- Recovery failure

Failures prioritize correctness over availability.

## 29. Trust Boundary

Only the State Manager may declare current canonical state.

Runtime output is observation.

Cache is acceleration.

Memory is temporary context.

Events are historical records.

Canonical current truth belongs to the State Manager.

---

## 30. Architectural Invariants

1. Every canonical state belongs to a registered object.
2. Every state mutation is versioned.
3. Every transition has an identity and audit context.
4. Every mutation passes policy validation.
5. Every mutation occurs inside a transaction boundary.
6. Every accepted result is persisted.
7. Cache never overrides canonical state.
8. AI never directly mutates canonical state.
9. Failed transitions do not partially commit.
10. Recovery restores only validated state.

---

## 31. Summary

The Kernel State Manager is the canonical authority for current system state in cmdOS.

It governs state creation, reads, transitions, versioning, persistence, reconciliation and recovery.

Every mutation is explicit, validated, transactional, observable and auditable.

Canonical truth belongs to the State Manager.

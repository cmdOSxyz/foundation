# 06.03 Repository Structure

## Overview

The Repository Structure defines how cmdOS source code, documentation, modules, tools,
and development resources are organized. This document is the **single source of truth**
for the repository layout; `README.md` mirrors the tree in Section 4.

A well-designed structure lets cmdOS scale as an AI Execution Operating System while
keeping clear ownership, stable interfaces, and independent evolution of each component.

Core principle:

```
Repository → Organization → Modules → Components → Development Workflow
```

---

# 1. Purpose

Repository Structure establishes a consistent engineering organization model, enabling:

- Clear separation of responsibilities
- Better collaboration
- Faster development cycles
- Easier maintenance
- Independent component evolution
- Long-term scalability

---

# 2. Design Philosophy

cmdOS follows a modular, responsibility-driven design. Each component should have a clear
purpose, independent ownership, stable interfaces, and independent testing and evolution.

The structure prevents unclear ownership, duplicate implementations, tight coupling,
uncontrolled dependencies, and difficult maintenance.

---

# 3. Architecture Model

The repository is organized by system responsibility:

- **Core System Layer** — `kernel/`, `runtime/`: the execution foundation.
- **Intelligence Layer** — `ai/`, `providers/`: reasoning, routing, and provider access.
- **Agent Layer** — `agents/`: autonomous execution entities.
- **Extension Layer** — `capabilities/`, `plugins/`, `sdk/`: ecosystem expansion.
- **Application Layer** — `apps/`: user-facing surfaces.
- **Platform Layer** — `services/`, `schemas/`: supporting services and contracts.
- **Infrastructure Layer** — `infrastructure/`, `tools/`, `tests/`: operations and quality.

---

# 4. High-Level Repository Structure

```text
cmdOS/
├── docs/            # Specification (RFCs). Single source of truth for architecture.
├── apps/            # desktop/ (primary), mobile/, cli/, web/ (non-primary)
├── kernel/          # core/ scheduler/ memory/ security/ resources/ interfaces/
├── runtime/         # execution/ workflow/ scheduler/ process/ monitoring/
├── ai/              # router/ models/ reasoning/ context/ memory/ inference/
├── agents/          # core/ lifecycle/ planning/ memory/ communication/ templates/
├── capabilities/    # core/ registry/ built-in/ external/ examples/
├── plugins/         # loader/ manifest/ signing/ validation/ registry/
├── providers/       # AI provider adapters + external service integrations
├── sdk/             # agent-sdk/ capability-sdk/ plugin-sdk/ api/
├── services/        # identity/ sync/ marketplace/ billing/ analytics/
├── schemas/         # canonical data schemas / IDL (interface contracts)
├── infrastructure/  # cloud/ deployment/ containers/ monitoring/ automation/
├── tools/           # build/ generators/ debugging/ scripts/
└── tests/           # unit/ integration/ system/ security/ performance/
```

Each directory represents a major system responsibility.

---

# 5. Documentation — `docs/`

All technical knowledge and project information.

Responsibilities: architecture, development, product, business, and research
documentation; developer guides.

```
docs/
  01-vision/ 02-philosophy/ 03-design-principles/ 04-terminology/
  05-architecture/ 06-development/ 07-product/ 08-business/
  09-roadmap/ 10-appendix/
```

---

# 6. Kernel — `kernel/`

The lowest-level system foundation and the single execution authority.

Responsibilities: resource control, execution control, security enforcement, hardware
abstraction.

```
kernel/ core/ scheduler/ memory/ security/ resources/ interfaces/
```

---

# 7. Runtime — `runtime/`

The execution environment for approved operations.

Responsibilities: workflow execution, task scheduling, process management, execution
lifecycle.

```
runtime/ execution/ workflow/ scheduler/ process/ monitoring/
```

---

# 8. AI — `ai/`

Intelligence systems for reasoning and decision support.

Responsibilities: model management, AI routing, context processing, reasoning, memory
integration, inference.

```
ai/ router/ models/ reasoning/ context/ memory/ inference/
```

---

# 9. Agents — `agents/`

The framework and components for building autonomous agents.

Responsibilities: agent creation, lifecycle, planning, communication, memory.

```
agents/ core/ lifecycle/ planning/ memory/ communication/ templates/
```

---

# 10. Capabilities — `capabilities/`

Executable abilities that Agents invoke. Capabilities are the core execution primitive
and the only thing an Agent invokes at runtime.

Responsibilities: capability definitions, registration, execution, external integrations.

```
capabilities/ core/ registry/ built-in/ external/ examples/
```

---

# 11. Plugins — `plugins/`

Plugin packaging, validation, and installation. A Plugin is a distribution package that
registers Capabilities/Agents into the Registry after security validation. Agents invoke
Capabilities, never Plugins. See `docs/04-terminology/04.00-canonical-glossary.md` and
the Capability chapter (`05.3.*`).

Responsibilities: manifest and packaging, signing and provenance, validation and
security, installed-plugin registry, Capability/Agent registration on install.

```
plugins/ loader/ manifest/ signing/ validation/ registry/
```

Note: the developer-facing Plugin SDK lives in `sdk/plugin-sdk/`; plugin distribution
(discovery/install) is served by `services/marketplace/`.

---

# 12. SDK — `sdk/`

Tools for developers building extensions.

Responsibilities: developer APIs, Agent SDK, Capability SDK, Plugin SDK, examples.

```
sdk/ agent-sdk/ capability-sdk/ plugin-sdk/ api/ examples/
```

---

# 13. Providers — `providers/`

AI provider adapters and external service integrations behind a normalized interface.

Responsibilities: provider adapters (normalized request/response), external service
integrations, credential delegation to the kernel Secret Manager.

```
providers/ ai/ external/ adapters/
```

---

# 14. Applications — `apps/`

User-facing cmdOS applications.

Responsibilities: desktop (primary), mobile, command-line, and a non-primary web surface.

```
apps/ desktop/ mobile/ cli/ web/
```

---

# 15. Services — `services/`

Backend and platform services.

Responsibilities: identity, synchronization, marketplace (distributes Plugins), billing,
analytics.

```
services/ identity/ sync/ marketplace/ billing/ analytics/
```

---

# 16. Schemas — `schemas/`

Canonical, machine-readable interface contracts (data schemas / IDL). Human-readable
specifications live in `docs/`.

Responsibilities: object/command schemas, intent and execution-plan schemas, capability
and permission contracts, event schemas.

```
schemas/ objects/ commands/ intent/ capabilities/ events/
```

---

# 17. Tools — `tools/`

Developer and operational utilities, including automation scripts.

```
tools/ build/ generators/ debugging/ scripts/
```

---

# 18. Tests — `tests/`

Quality assurance systems.

```
tests/ unit/ integration/ system/ security/ performance/
```

---

# 19. Infrastructure — `infrastructure/`

Deployment and operational systems.

```
infrastructure/ cloud/ deployment/ containers/ monitoring/ automation/
```

---

# 20. Development Workflow

```
Feature Request → Design → Development Branch → Implementation
→ Testing → Code Review → Merge → Release
```

---

# 21. Branch Strategy

- **Main** — production-ready code.
- **Development** — integration branch for active development.
- **Feature** — individual features (e.g. `feature/agent-memory`).
- **Release** — preparing software releases.

---

# 22. Naming Standards

Use clear, consistent, responsibility-based names — for example `cmdos-kernel`,
`cmdos-runtime`, `cmdos-agent-sdk`, `cmdos-plugin-sdk`. Avoid `test1`, `new-folder`,
`temp`.

---

# 23. Monorepo and Multi-Repository Strategy

cmdOS uses a hybrid model. The core system (kernel, runtime, core AI) is a monorepo for
coordination and shared dependencies. The ecosystem (plugins, external agents, community
projects) uses multiple repositories for independent development and flexible ownership.

---

# 24. Design Principles

Organized, modular, scalable, developer-friendly, maintainable.

---

# Summary

Repository Structure defines how cmdOS engineering resources are organized. Documentation
explains the system; the Kernel provides the foundation; Runtime manages execution; AI
provides intelligence; Agents perform autonomous tasks; Capabilities provide actions;
Plugins package and distribute them; the SDK enables developers; Applications connect
users; Services and Schemas support the platform; Infrastructure supports production.

Together they form the engineering foundation of the AI Execution Operating System.

# RFC-0012: aipc — Agent IPC (MCP-Style Tool Surface)

Version: 1.0
Status: Accepted
Category: Architecture (services)
Author: Lead Architect
Depends on: RFC-0004 (Object Model), RFC-0010 (Policy)
Implemented by: `services/aipc`

---

# 1. Summary

AIPC is how a user agent reaches capabilities. Every capability publishes its
actions as MCP-style **tools**; the agent asks AIPC to invoke a tool, and **every
call is routed through the policy gate (Alios) before it can run**. MCP is the
shape of the tool surface; the permission gate is the enforcement. This is the OS
ABI for tools.

# 2. Scope of this RFC

This crate is the in-process **spine** of AIPC: the tool registry and the
permission-gated router. The wire transport to *external* MCP servers
(stdio / JSON-RPC) plugs into this same interface later without changing the
router or the gate — Interfaces first, Runtime second.

# 3. Design

- `Tool { capability, name, description, parameters, risk }` — the MCP-style
  descriptor an agent sees; `qualified_name()` is `capability.action`.
- `Aipc` registry: `register`, `list_tools`, `tool(cap, action)`.
- `ToolCall { capability, action, arguments, spend }` — an agent's invocation.
- `route(call, mandate, budget)` runs the call through `cmd-policy` and returns:
  `Authorized { tool }`, `NeedsApproval { reason }`, `Blocked { reason }`, or
  `UnknownTool`. AIPC authorizes; it never executes — execution goes through the
  kernel + transaction engine, staying reversible and recorded.

# 4. Load-Bearing Property

A call is gated **even when the tool exists**: a registered spend tool with no
budget is still blocked. AIPC provides no path around policy. This is the point
where the 200+ MCP-server ecosystem becomes usable *safely* — every third-party
tool call is kernel-mediated.

# 5. Testing

7 tests, all green, no warnings: register/list; read-only authorized without
authority; unknown tool; R1-within-mandate authorized; blocked outside mandate;
R3 needs approval; and gate-enforced-even-when-tool-exists.

# 5b. First-party capability catalog (added)

`aipc::catalog` declares MCP-style tools for the first-party capabilities:
`filesystem_tools()` (list/read/rename/move/delete) and `browser_tools()`
(navigate/read/screenshot/fill/submit/click_buy/pay), each with the risk class
matching the capability's own `risk_of`. `Aipc::new().with_first_party()`
registers them all, giving the agent a populated tool surface at startup.

Declaring tools here (not in the capabilities) keeps capabilities free of any
dependency on AIPC — no cycle. The catalog is the single binding point between
capability actions and the tool surface. Browser submit/click_buy/pay are
registered R3, so routing them requires human approval.

# 6. Next

external MCP servers; then the agent/planning layer that turns intents into tool

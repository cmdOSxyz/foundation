# Systems Should Explain Themselves

Version: 1.0
Status: Canonical
Category: Foundation

---

# 1. Purpose

This document establishes the seventh philosophical principle of cmdOS:

> Every execution should be explainable.

Users should never be expected to trust a system they cannot understand.

Every significant action performed by cmdOS shall be accompanied by sufficient explanation to allow humans to understand what happened, why it happened, and how it happened.

Explainability is a fundamental requirement for trusted execution.

---

# 2. Why Explainability Matters

Execution changes the world.

An execution may:

- transfer money
- deploy infrastructure
- modify data
- execute code
- control devices
- access confidential information

Such actions require transparency.

Users should never have to guess why the system behaved as it did.

---

# 3. Explainability Is Different from Transparency

Transparency exposes information.

Explainability organizes information into understandable reasoning.

Example:

Transparent

```
Policy 4.2 rejected execution.
```

Explainable

```
Execution was rejected because
the requested action exceeds the user's
financial approval limit defined by Policy 4.2.
```

Understanding—not merely visibility—is the objective.

---

# 4. Levels of Explanation

cmdOS recognizes multiple explanation levels.

### User Level

Simple and concise.

Example:

> Permission required to continue.

---

### Technical Level

Includes execution details.

Example:

- policy matched
- capability selected
- execution plan
- dependency resolution

---

### Administrative Level

Explains governance decisions.

Example:

- approval chain
- compliance rules
- organization policy
- audit evidence

---

### Developer Level

Includes implementation information.

Example:

- runtime logs
- API calls
- execution graph
- stack traces
- timing information

Each audience receives explanations appropriate to its responsibilities.

---

# 5. Explainable Planning

Execution plans shall be explainable before execution begins.

Users should understand:

- intended outcome
- selected capabilities
- estimated impact
- required permissions
- expected side effects
- recovery options

Planning should never appear mysterious.

---

# 6. Explainable Decisions

AI-generated decisions should include reasoning.

Reasoning may include:

- objectives considered
- constraints evaluated
- policies applied
- alternatives rejected
- confidence indicators
- supporting evidence

The explanation should enable informed human review.

---

# 7. Explainable Execution

During execution, cmdOS shall explain:

- current step
- completed steps
- remaining work
- encountered issues
- execution progress
- state transitions

Execution should remain observable throughout its lifecycle.

---

# 8. Explainable Failures

Failures require better explanations than success.

Every failure should answer:

- what failed
- where it failed
- why it failed
- what data was affected
- whether recovery occurred
- recommended next actions

A failure without explanation reduces trust.

---

# 9. Explainable Recovery

Recovery actions shall also be explained.

Example:

```
Deployment failed.

Rollback initiated.

Database restored.

Service health verified.

Execution returned to previous state.
```

Recovery should never occur silently.

---

# 10. Explainability and Trust

Explainability strengthens trust.

Trust grows when users understand:

- intentions
- decisions
- execution
- recovery

Trust decreases when behavior appears arbitrary.

Explainability therefore contributes directly to trusted execution.

---

# 11. Explainability and Audit

Audit records preserve historical evidence.

Explainability interprets that evidence.

Audit answers:

"What happened?"

Explainability answers:

"Why did it happen?"

Both are required.

---

# 12. Explainability and Privacy

Explainability must respect privacy.

Explanations shall never reveal:

- unauthorized information
- confidential credentials
- protected personal data
- secret organizational policies

The amount of explanation is constrained by authorization.

---

# 13. Explainability by Design

Explainability is not an optional feature.

It must be designed into:

- planning
- execution
- monitoring
- recovery
- governance
- security
- auditing

Retrofitted explanations are incomplete.

---

# 14. Architectural Consequences

Explainability requires dedicated platform components.

Including:

- Explainability Engine
- Decision Trace Framework
- Execution Timeline
- Reasoning Recorder
- Explanation Generator
- Audit Correlator
- Evidence Store
- User Explanation API

These components collectively provide explainable execution.

---

# 15. Design Principles

Explainable systems follow these principles.

### Explain Before Execute

Users should understand significant actions before approval.

---

### Explain While Executing

Execution progress remains understandable.

---

### Explain After Completion

Results should summarize what occurred.

---

### Explain Failures Completely

Failures require complete explanations.

---

### Explain Without Oversharing

Only authorized information is revealed.

---

# 16. Explainability Invariants

The following statements shall always remain true.

1. Significant actions are explainable.
2. Plans are explainable.
3. Decisions are explainable.
4. Failures are explainable.
5. Recovery is explainable.
6. Explanations respect permissions.
7. Explainability strengthens trust.
8. Explainability is built into architecture.

These invariants apply throughout the cmdOS platform.

---

# 17. Relationship to the Specification

This document establishes the explainability philosophy of cmdOS.

Future specifications—including Explainability Framework, Planning Engine, Execution Runtime, Audit Framework, Observability Platform, Security Architecture, and AI Execution Protocol—shall implement these principles.

Explainability is considered a mandatory property of trusted execution.

---

# 18. Summary

Execution should never be a black box.

Every important action should be understandable.

Every important decision should be justifiable.

Every important failure should be explainable.

By making reasoning visible and understandable, cmdOS enables humans to maintain confidence, oversight, and accountability while delegating increasingly complex work to AI.

Explainability transforms execution into trusted execution.

# Execution Requires Context

Version: 1.0
Status: Canonical
Category: Foundation

---

# 1. Purpose

This document establishes the sixth philosophical principle of cmdOS:

> Intent without context cannot produce reliable execution.

Intent defines the desired outcome.

Context determines the meaning of that outcome.

Reliable execution requires both.

---

# 2. Why Context Matters

Humans naturally rely on context.

The same sentence may have completely different meanings depending on:

- time
- location
- identity
- environment
- previous actions
- organizational policies

Computers traditionally require users to specify every detail explicitly.

cmdOS instead reconstructs the missing context before execution begins.

---

# 3. Intent Alone Is Insufficient

Consider the instruction:

> Open the report.

Questions immediately arise.

- Which report?
- Stored where?
- Latest version?
- Shared by whom?
- Read-only or editable?
- Does the user have permission?

Intent identifies the objective.

Context identifies the target.

---

# 4. Context Defines Meaning

Meaning is produced by combining intent with context.

```
Intent

+

Context

↓

Meaning

↓

Execution
```

Without context, identical intent may produce different outcomes.

---

# 5. Sources of Context

Context may originate from many sources.

Examples include:

Human Context

- identity
- preferences
- language
- timezone
- responsibilities

Environmental Context

- device
- operating system
- location
- network
- battery status

Organizational Context

- department
- role
- policies
- security classification
- approval chain

Execution Context

- active workflow
- previous steps
- execution history
- temporary variables

Knowledge Context

- documents
- memory
- retrieved information
- enterprise knowledge
- previous conversations

Context is cumulative.

---

# 6. Context Is Dynamic

Context continuously changes.

Examples:

Morning

↓

Working Hours

↓

Travel

↓

Meeting

↓

Offline

↓

Emergency

The correct execution strategy may change even though the user's intent remains identical.

---

# 7. Context Before Planning

Planning without context produces unreliable execution.

The planning sequence shall therefore be:

```
Intent

↓

Context Resolution

↓

Planning

↓

Validation

↓

Execution
```

Planning must never begin with incomplete context when correctness depends upon it.

---

# 8. Context Resolution

Context resolution is the process of gathering relevant information required to understand intent.

Possible sources include:

- user profile
- organizational knowledge
- execution history
- runtime state
- connected services
- external systems
- memory
- environmental sensors

Context should be collected only when necessary.

---

# 9. Context Is Not Memory

Memory stores information over time.

Context represents the information relevant to a specific execution.

Memory contributes to context.

It does not replace it.

```
Memory

↓

Relevant Information

↓

Context

↓

Execution
```

---

# 10. Context Is Not Knowledge

Knowledge represents facts.

Context represents applicability.

Example:

Knowledge:

> The meeting room exists.

Context:

> The meeting room is currently occupied.

Knowledge alone cannot determine execution.

---

# 11. Context and Permissions

Context must never bypass governance.

Examples:

Knowing a password does not grant authority.

Knowing a document exists does not authorize access.

Context informs execution.

Permissions authorize execution.

These concerns remain separate.

---

# 12. Context Quality

Reliable execution depends upon context quality.

Context should be:

- relevant
- current
- verified
- consistent
- explainable
- permission-aware

Poor context produces poor execution.

---

# 13. Context Minimization

cmdOS shall collect only the context required for successful execution.

The platform avoids unnecessary data collection.

Benefits include:

- improved privacy
- reduced complexity
- lower execution cost
- better performance
- smaller attack surface

Context should always be proportional to the requested objective.

---

# 14. Architectural Consequences

Execution requiring context introduces dedicated architectural components.

Including:

- Context Engine
- Context Resolver
- Context Store
- Context Cache
- Context Policy Engine
- Context Providers
- Context Validator
- Context Version Manager

These components collectively form the Context Model.

---

# 15. Design Principles

Context-aware execution follows these principles.

### Resolve Before Execute

Required context is gathered before execution begins.

---

### Collect Only What Is Necessary

Context collection follows the principle of minimal sufficiency.

---

### Separate Context from Intent

Intent defines purpose.

Context defines meaning.

---

### Context Evolves

Execution adapts as context changes.

---

### Context Respects Governance

Context never overrides permissions or policies.

---

# 16. Context Invariants

The following statements shall always remain true.

1. Intent requires context.
2. Context precedes planning.
3. Context is dynamic.
4. Context is execution-specific.
5. Context never replaces permissions.
6. Context collection is minimal.
7. Context remains explainable.
8. Context improves execution accuracy.

These invariants apply across the cmdOS platform.

---

# 17. Relationship to the Specification

This document establishes the context-aware philosophy of cmdOS.

Future specifications—including Context Model, Memory Architecture, Knowledge Architecture, Planning Engine, Workflow Runtime, Agent Runtime, and Kernel—shall implement the principles defined here.

Context becomes a first-class architectural object alongside Intent.

---

# 18. Summary

Intent defines what should be achieved.

Context explains what that intent means in the current situation.

Execution succeeds only when both are understood.

cmdOS therefore treats context not as optional metadata, but as an essential component of trusted execution.

# Security by Default

Version: 1.0
Status: Canonical
Category: Foundation

---

# 1. Purpose

This document establishes the sixth engineering principle of cmdOS:

> Every component shall be secure by default.

Security is not an optional feature.

It is a foundational architectural property.

Every execution, capability, runtime, plugin, API, and agent shall begin from a secure state rather than requiring security to be added later.

---

# 2. Definition

Security by Default means that the safest behavior is the default behavior.

Without explicit authorization, execution shall not proceed.

The absence of configuration shall never reduce security.

---

# 3. Why Security Must Be Default

Execution systems interact with:

- user identities
- financial assets
- enterprise data
- operating systems
- cloud infrastructure
- blockchain networks
- AI providers

A single insecure default may compromise the entire execution platform.

Security therefore precedes convenience.

---

# 4. Default Deny

Every executable capability begins with:

```
Permission

↓

Denied
```

Access is granted only after successful evaluation.

cmdOS never assumes authority.

---

# 5. Least Privilege

Every component shall receive only the permissions required to perform its task.

Example:

Instead of:

```
Filesystem

↓

Full Access
```

Grant:

```
Read

Project Folder Only
```

Minimal authority reduces potential damage.

---

# 6. Explicit Authorization

Authorization must always be explicit.

Execution shall identify:

- requesting identity
- delegated authority
- applicable policy
- execution scope
- expiration

Implicit authorization is prohibited.

---

# 7. Zero Trust

cmdOS adopts a Zero Trust architecture.

No component is trusted solely because it resides within the same environment.

Every interaction requires verification.

Examples include:

- plugins
- AI providers
- agents
- cloud services
- desktop processes
- APIs

Trust is continuously evaluated.

---

# 8. Capability Isolation

Capabilities execute within controlled boundaries.

Isolation may include:

- permission boundaries
- process isolation
- runtime isolation
- filesystem restrictions
- network policies
- resource quotas

Isolation prevents failures from propagating.

---

# 9. Secure Defaults

Platform defaults shall favor safety.

Examples:

- plugins disabled until installed
- external execution disabled until approved
- destructive operations require confirmation
- sensitive permissions require approval
- audit logging enabled
- encryption enabled

Unsafe defaults are prohibited.

---

# 10. Secrets Management

Secrets shall never be:

- hardcoded
- embedded in source code
- stored in logs
- exposed to plugins unnecessarily

Secrets should be managed through dedicated secure storage.

Examples include:

- API keys
- passwords
- tokens
- certificates
- signing keys

---

# 11. Defense in Depth

Security shall exist at multiple architectural layers.

```
Identity

↓

Authentication

↓

Authorization

↓

Policy

↓

Execution

↓

Audit

↓

Recovery
```

No single layer is solely responsible for protection.

---

# 12. Secure Failure

When uncertainty exists, the secure behavior shall be selected.

Examples:

Instead of:

```
Permission Check Failed

↓

Continue
```

cmdOS performs:

```
Permission Check Failed

↓

Reject Execution
```

Failures should never accidentally increase authority.

---

# 13. Security and AI

AI does not receive implicit trust.

Every AI-generated action must pass:

- policy evaluation
- permission validation
- execution verification
- audit recording

AI recommendations never bypass governance.

---

# 14. Architectural Consequences

Security-first architecture requires dedicated platform components.

Including:

- Identity Framework
- Authentication Service
- Authorization Engine
- Permission Manager
- Policy Engine
- Secret Manager
- Trust Framework
- Audit Framework

Together, these components establish trusted execution.

---

# 15. Design Principles

Secure systems follow these principles.

### Deny by Default

Access begins in the denied state.

---

### Least Privilege

Components receive only necessary authority.

---

### Explicit Authorization

Authority is granted deliberately.

---

### Zero Trust

Verification is continuous.

---

### Secure Failure

Failures preserve security.

---

# 16. Security Invariants

The following statements shall always remain true.

1. Execution is denied by default.
2. Authority is explicitly granted.
3. Least privilege is enforced.
4. Zero Trust applies universally.
5. Secrets remain protected.
6. AI does not bypass governance.
7. Security precedes convenience.
8. Secure behavior is the default behavior.

These invariants apply throughout the cmdOS platform.

---

# 17. Relationship to the Specification

This document establishes the security-by-default engineering principle.

Future specifications—including Security Architecture, Identity Model, Permission Model, Policy Engine, Capability Framework, Plugin Framework, Runtime Architecture, and Kernel—shall implement these principles.

Security is considered a mandatory architectural property rather than an optional capability.

---

# 18. Summary

Security cannot be added after architecture is complete.

It must exist from the first design decision.

By adopting deny-by-default, least privilege, explicit authorization, Zero Trust, and secure failure handling, cmdOS creates an execution platform that remains trustworthy even as its capabilities, intelligence providers, and deployment environments evolve.

Security by Default is therefore a foundational engineering principle of the AI Execution Operating System.

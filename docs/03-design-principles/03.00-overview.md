# Chapter 03 — Design Principles

Version: 1.0

---

# 1. Purpose

This chapter defines the engineering principles that govern the design and implementation of cmdOS.

While Vision defines purpose and Philosophy defines mindset, Design Principles define how software is engineered.

These principles translate philosophy into practical architectural decisions.

---

# 2. From Philosophy to Engineering

Philosophy answers:

> What should we believe?

Design Principles answer:

> How should we build?

Every implementation decision should be traceable back to one or more philosophical principles established in Chapter 02.

---

# 3. Objectives

The principles in this chapter ensure that cmdOS remains:

- predictable
- maintainable
- extensible
- secure
- scalable
- observable
- recoverable
- provider-independent

They apply equally to every subsystem, from the Kernel to Plugins.

---

# 4. Design Hierarchy

Engineering decisions follow a strict hierarchy.

```
Vision

↓

Philosophy

↓

Design Principles

↓

Architecture

↓

Implementation

↓

Optimization
```

No lower layer may violate the intent of a higher layer.

---

# 5. Scope

The principles defined in this chapter apply to every architectural component, including:

- Kernel
- Runtime
- Agent Framework
- Capability System
- Workflow Engine
- Policy Engine
- SDK
- APIs
- Plugins
- Desktop Runtime
- Browser Runtime
- Cloud Runtime

No subsystem is exempt.

# 7. Relationship to the Specification

Every future RFC inherits these engineering principles.

Architectural proposals that violate these principles require explicit justification and formal review.

The burden of proof lies with the exception, not the principle.

---

# 8. Constitutional Authority

These principles are normative.

They are not recommendations.

Every implementation shall conform unless a future specification explicitly supersedes them.

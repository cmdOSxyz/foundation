# Explicit over Implicit

Version: 1.0
Status: Canonical
Category: Foundation

---

# 1. Purpose

This document establishes the third engineering principle of cmdOS:

> Every important behavior shall be explicit rather than implicit.

Hidden assumptions create unpredictable systems.

Explicit systems are easier to understand, validate, secure, test, and evolve.

cmdOS therefore favors clarity over convenience.

---

# 2. Definition

Explicitness means that every significant aspect of execution is declared rather than inferred.

This includes:

- intent
- permissions
- policies
- dependencies
- execution plans
- state transitions
- authority
- failure handling

Nothing critical should depend upon undocumented assumptions.

---

# 3. Why Explicitness Matters

Implicit behavior introduces uncertainty.

Examples include:

- hidden defaults
- automatic privilege escalation
- undocumented dependencies
- invisible state changes
- inferred permissions
- silent retries

These behaviors reduce trust because users cannot predict system behavior.

---

# 4. Explicit Intent

Execution begins with explicitly identified intent.

The system may infer user meaning during interpretation.

However, before execution begins, the resulting intent must be represented explicitly inside the execution pipeline.

Intent should never remain ambiguous.

---

# 5. Explicit Authority

Authority shall always be declared.

Never inferred.

Examples include:

- user identity
- delegated permissions
- approval grants
- execution scope
- organizational authority

Every execution must identify its source of authority.

---

# 6. Explicit Dependencies

Every dependency shall be visible.

Examples include:

- required capabilities
- external services
- plugins
- AI providers
- runtime components
- execution prerequisites

Hidden dependencies increase operational risk.

---

# 7. Explicit State

State transitions shall always be declared.

Instead of:

```
Running

↓

Completed
```

cmdOS prefers:

```
Created

↓

Validated

↓

Authorized

↓

Executing

↓

Verifying

↓

Completed
```

Each transition becomes observable.

---

# 8. Explicit Policies

Policies shall be evaluated explicitly.

Execution should never rely upon undocumented organizational assumptions.

Every policy decision should identify:

- matching policy
- decision outcome
- evaluation timestamp
- responsible authority

Policy behavior must remain reproducible.

---

# 9. Explicit Contracts

Every architectural component shall define an explicit contract.

Contracts include:

- purpose
- inputs
- outputs
- permissions
- guarantees
- failure modes
- compatibility
- version

Contracts eliminate ambiguity between components.

---

# 10. Explicit Failure

Failures shall never remain hidden.

Every failure should explicitly identify:

- affected component
- reason
- severity
- recovery strategy
- current execution state

Silent failures are prohibited.

---

# 11. Explicit Defaults

Defaults are useful but must remain visible.

Example:

Instead of silently selecting a provider,

cmdOS records:

```
Default Provider

↓

Provider-A

↓

Selection Reason

↓

Organization Policy
```

Users should understand why defaults were chosen.

---

# 12. Explicit Configuration

Configuration should always be inspectable.

Examples include:

- runtime configuration
- capability configuration
- AI routing configuration
- execution policies
- security settings
- organizational preferences

Hidden configuration creates unpredictable systems.

---

# 13. Explicit AI Decisions

AI-generated recommendations shall be represented explicitly.

Execution records should include:

- selected model
- reasoning summary
- confidence indicators
- applied constraints
- selected capability

AI reasoning must not become an invisible dependency.

---

# 14. Architectural Consequences

Explicit architecture requires dedicated mechanisms.

Including:

- Execution Graph
- State Manager
- Contract Registry
- Policy Engine
- Configuration Registry
- Audit Framework
- Dependency Resolver
- Explainability Engine

These components ensure explicit execution behavior.

---

# 15. Design Principles

Explicit systems follow these principles.

### Declare Everything Important

Critical behavior should never be hidden.

---

### Make Dependencies Visible

Dependencies should always be inspectable.

---

### Record Every Decision

Important decisions become execution records.

---

### Reveal State

Execution state remains observable.

---

### Avoid Hidden Magic

Convenience must never reduce understanding.

---

# 16. Explicitness Invariants

The following statements shall always remain true.

1. Authority is explicit.
2. Dependencies are explicit.
3. State transitions are explicit.
4. Policies are explicit.
5. Failures are explicit.
6. Configuration is explicit.
7. AI decisions are traceable.
8. Hidden behavior is minimized.

These invariants apply throughout the cmdOS platform.

---

# 17. Relationship to the Specification

This document establishes the explicit-first engineering principle.

Future specifications—including State Model, Capability Contracts, Execution Graph, Policy Engine, Audit Framework, Explainability Framework, and Kernel—shall prefer explicit declarations over implicit behavior.

Whenever implicit behavior cannot be avoided, it shall be documented, observable, and governed.

---

# 18. Summary

Clarity is a prerequisite for trust.

Explicit systems are easier to understand, secure, debug, audit, and evolve.

By making authority, intent, dependencies, state, and execution behavior explicit, cmdOS minimizes ambiguity and establishes a predictable foundation for trusted AI execution.

Explicitness is therefore a fundamental engineering principle of the AI Execution Operating System.

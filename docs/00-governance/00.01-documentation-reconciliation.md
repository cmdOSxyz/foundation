# RFC-0001 — Documentation Reconciliation

Version: 1.0
Status: Proposed (awaiting maintainer approval)
Category: Governance
Supersedes: conflicting definitions in `README.md`, `ROADMAP.md`, `docs/06-development/06.03`, and scattered security/terminology docs
Author: Lead Architect

---

## 0. Purpose

The cmdOS specification has grown faster than it has been integrated. Six structural
inconsistencies now exist across the documentation set. Each one, left unresolved,
produces cross-reference ambiguity and lets independent documents drift apart.

This RFC does one thing: it makes each conflict resolve to a **single source of truth**.
It follows the project's own rule — *documentation first, RFC before implementation* —
and changes no code. Every decision below is reversible via a follow-up RFC, but once
approved it becomes canonical and all other documents must conform.

The six conflicts:

| # | Conflict | Canonical decision |
|---|----------|--------------------|
| 1 | `05.xx` numbering collisions across architecture sub-trees | Hierarchical `05.S.NN` scheme |
| 2 | Two competing roadmaps (Phases 0–9 vs Stages 0–4) | Stages are canonical; Phases become workstreams mapped into Stages |
| 3 | Two competing repository structures | One canonical top-level layout |
| 4 | Plugin vs Capability overlap | Capability = execution primitive; Plugin = distribution package |
| 5 | Security documented in six places with no owner | One canonical architecture; others become references |
| 6 | Terminology drift (Admin Runtime / Interaction Layer / etc.) | Canonical glossary with deprecated aliases |

---

## 1. Conflict 1 — Document Numbering

### 1.1 Problem

Chapter 05 (Architecture) contains seven sub-trees: `ai/`, `agent/`, `capability/`,
`communication/`, `runtime/`, `security/`, and `kernel/core/`. Every sub-tree restarts
its numbering at `05.0x`, so a single number maps to several unrelated documents:

- `05.02` = AI Router **and** Agent Lifecycle **and** Capability Registry **and** State Manager
- `05.07` = Capability Summary **and** Communication Summary **and** Security Summary
- Kernel core files are named `05.01`–`05.54` but carry a stale internal heading
  `# Part 10.35.NN`, evidence of a renumbering that was never propagated.

### 1.2 Decision

Adopt a **three-level hierarchical scheme**: `CC.S.NN`.

- `CC` — chapter (`05` = Architecture)
- `S` — sub-domain within the chapter (single digit)
- `NN` — document within the sub-domain (two digits, zero-padded)

Sub-domain assignments for Chapter 05:

| `S` | Sub-domain | Directory |
|-----|------------|-----------|
| 1 | AI | `docs/05-architecture/1-ai/` |
| 2 | Agent | `docs/05-architecture/2-agent/` |
| 3 | Capability | `docs/05-architecture/3-capability/` |
| 4 | Communication | `docs/05-architecture/4-communication/` |
| 5 | Runtime | `docs/05-architecture/5-runtime/` |
| 6 | Security | `docs/05-architecture/6-security/` |
| 7 | Kernel Core | `docs/05-architecture/7-kernel/core/` |

Rules:

1. A document number is globally unique. `05.1.02` and `05.2.02` are different files;
   `05.02` alone is no longer valid.
2. Each sub-domain reserves `NN = 00` for its `README`/overview and its highest `NN`
   for its `-summary` document.
3. The `# Part 10.35.NN` headings in kernel files are deleted; the H1 of each kernel
   file must equal its canonical number (e.g. `# 05.7.03 Command Manager`).
4. Chapters other than 05 already use `CC.NN` and are unaffected, except that any
   cross-reference into Chapter 05 must use the new `05.S.NN` form.

## 2. Conflict 2 — Roadmap

### 2.1 Problem

Two roadmaps describe different axes but read as competing timelines:

- `ROADMAP.md` (root): **Phases 0–9** — Foundation, Kernel, Execution Engine, Runtime,
  Capability System, AI Integration, Desktop Agent, Distributed, SDK, Enterprise.
  This is an *engineering build sequence*.
- `docs/09-roadmap`: **Stages 0–4** — Foundation, MVP, Agent Platform, Execution OS,
  AI-Native OS. This is a *product / strategic sequence*.
- `CLAUDE.md`: a desktop-first *component build order* (Shell → Control Center →
  Agent Runtime → Execution Engine → Capability System → AI Router → Memory).

These are three different things presented as three timelines.

### 2.2 Decision

Establish **one roadmap with two explicit axes**:

- **Stages 0–4 are the canonical product roadmap.** They are outcome-based and belong
  at the top level. `docs/09-roadmap` is the single source of truth.
- **Phases become engineering workstreams**, not a timeline. Each Phase delivers into
  one or more Stages. `ROADMAP.md` at the repo root is reduced to a short pointer that
  links to `docs/09-roadmap` and shows the mapping table below.
- The `CLAUDE.md` component build order is scoped as **the sequencing inside Stage 1 (MVP)**
  and is documented there, not as a third roadmap.

### 2.3 Phase → Stage mapping

| Engineering workstream (Phase) | Delivers into Stage |
|-------------------------------|---------------------|
| P0 Foundation | Stage 0 Foundation |
| P1 Kernel | Stage 1 MVP |
| P2 Execution Engine | Stage 1 MVP |
| P3 Runtime | Stage 1 MVP → Stage 3 Execution OS |
| P4 Capability System | Stage 1 MVP → Stage 2 Agent Platform |
| P5 AI Integration | Stage 1 MVP |
| P6 Desktop Agent | Stage 3 Execution OS |
| P7 Distributed cmdOS | Stage 3 Execution OS → Stage 4 AI-Native OS |
| P8 SDK | Stage 2 Agent Platform |
| P9 Enterprise | Stage 4 AI-Native OS |

The "Phase" label is retained only as a workstream identifier. It never appears as a
standalone timeline again.

## 3. Conflict 3 — Repository Structure

### 3.1 Problem

Two incompatible layouts exist:

- `README.md`: `kernel/ specs/ schemas/ sdk/ runtime/ providers/ capabilities/ tests/ tools/ scripts/`
- `docs/06-development/06.03`: `capabilities/ plugins/ sdk/ apps/ services/ infrastructure/ tools/ tests/`

Each omits directories the other treats as core.

### 3.2 Decision

One canonical top-level layout. `docs/06-development/06.03` becomes the single source
of truth; the `README.md` tree is replaced by a link to it plus this diagram.

```text
cmdOS/
├── docs/            # Specification (RFCs). Single source of truth for architecture.
├── apps/            # User-facing applications
│   ├── desktop/     #   Primary product (Windows / macOS / Linux)
│   ├── mobile/      #   Companion product
│   └── cli/         #   Command-line surface
├── kernel/          # Kernel core (objects, state, commands, planner, engine, ...)
├── runtime/         # Runtime environments (desktop, browser, containers)
├── capabilities/    # Capability system + first-party capabilities (see Conflict 4)
├── providers/       # AI provider adapters + external service integrations
├── sdk/             # agent-sdk / capability-sdk / plugin-sdk / api
├── services/        # Supporting cloud services: identity, sync, marketplace, billing
├── schemas/         # Canonical data schemas / IDL (interface contracts)
├── infrastructure/  # Deployment, containers, monitoring, automation
├── tools/           # Build, generators, debugging, scripts
└── tests/           # unit / integration / system / security / performance
```

Reconciliation notes:

1. `specs/` and `schemas/` are merged into **`schemas/`** (machine-readable interface
   contracts). Human-readable specs live in `docs/`.
2. `scripts/` is folded into **`tools/`**.
3. `web/` is dropped as a top-level app because the product direction states the primary
   experience is not browser-based; a web surface, if built, lives under `apps/web/` as
   a non-primary target.
4. There is **no top-level `plugins/`** — see Conflict 4. Plugin packaging lives in
   `sdk/plugin-sdk/`; plugin distribution lives in `services/marketplace/`.

## 4. Conflict 4 — Plugin vs Capability

### 4.1 Problem

Both concepts have a registry, an SDK, a marketplace, and a security model, and their
responsibilities overlap. The relationship is never stated. The only concrete clue is a
flow already present in the security docs:

```
Plugin → Identity Verification → Security Approval → Capability Registration → Execution
```

That flow implies plugins *produce* capabilities. This RFC makes that explicit.

### 4.2 Decision

**Capability is the core execution primitive. Plugin is the distribution package that
delivers one or more Capabilities (and optionally Agents) into cmdOS.**

Definitions:

- **Capability** — a versioned interface contract plus an implementation that an Agent
  invokes to act on the world (e.g. `email.send`, `browser.open`). A first-class kernel
  concept, held in the **Capability Registry**, governed by permissions and policy.
- **Plugin** — a signed, versioned bundle (`manifest + code`) that a developer publishes.
  On install, after security validation, a Plugin **registers** its Capabilities and/or
  Agents into the Registry. A Plugin is packaging and provenance, not a runtime concept.

Consequences (removes the duplication):

| Concern | Canonical owner |
|---------|-----------------|
| Ability contract + execution | Capability (Capability Registry, Capability Runtime) |
| Packaging, signing, provenance, versioned distribution | Plugin (Plugin SDK) |
| Discovery & install of third-party bundles | Marketplace (one marketplace, delivers Plugins) |
| What an Agent can invoke at runtime | Capability Registry only |
| Building an ability | Capability SDK |
| Building an autonomous worker | Agent SDK |
| Packaging any of the above for publish | Plugin SDK (wraps the other two) |

Canonical lifecycle:

```text
Developer
   ↓  (Capability SDK / Agent SDK)
Capability / Agent implementation
   ↓  (Plugin SDK: package + sign)
Plugin bundle
   ↓  publish
Marketplace
   ↓  install + security validation
Capability Registration
   ↓
Agent invocation → Runtime → Kernel → Execution
```

Documentation actions:

1. Any doc describing a standalone "Plugin runtime" or "Plugin registry" that an Agent
   invokes directly is corrected: Agents invoke **Capabilities**, never Plugins.
2. "Plugin marketplace" and "Capability marketplace" are unified into one **Marketplace**
   that distributes Plugins.
3. The Capability Architecture chapter (`05.3.*`) gains one paragraph defining the
   Plugin→Capability relationship above and links to the Plugin SDK spec.

## 5. Conflict 5 — Security Source of Truth

### 5.1 Problem

Security is specified in six places with no declared owner: `SECURITY.md`,
`04.10 Security Model`, the `05.6.*` security tree, kernel `05.7.19 Security Manager`,
`05.5.06 Runtime Security`, and `10.04 Security Reference`. The model is consistent today,
but nothing prevents future drift.

### 5.2 Decision

Assign each document an explicit role and a single canonical owner. No document is
deleted; each gains a `Canonical source:` header pointing at the owner.

| Document | Role after this RFC |
|----------|---------------------|
| `docs/05-architecture/6-security/` | **CANONICAL** — the security architecture and model. Single source of truth. |
| `SECURITY.md` (root) | Repo hygiene only: vulnerability reporting, supported versions, disclosure policy. Contains **no** architecture. |
| `04.10 Security Model` (terminology) | Definitions of security terms only; links to canonical for the model. |
| `10.04 Security Reference` (appendix) | Quick-reference index; links to canonical; holds no independent model. |
| `05.7.19 Security Manager` (kernel) | Subsystem spec scoped to kernel enforcement; references canonical. |
| `05.5.06 Runtime Security` | Subsystem spec scoped to the runtime layer; references canonical. |

Rule: any security statement that is not in the canonical tree must either match it or be
reduced to a link. The canonical model remains the pipeline already documented:

```
Identity → Permission → Policy → Isolation → Execution → Monitoring → Audit
```

## 6. Conflict 6 — Terminology

### 6.1 Problem

The same concept appears under multiple names across documents.

### 6.2 Decision — Canonical glossary

These become the canonical terms in Chapter 04. Deprecated aliases must be replaced on
next edit of each affected document.

| Canonical term | Meaning | Deprecated aliases |
|----------------|---------|--------------------|
| **Control Center** | The user-facing management/admin surface (agents, models, permissions, memory, logs, security). | Admin Desktop, Admin Control Center |
| **Interaction Layer** | The input/UI surface that captures user requests. | — |
| **Intent** | The structured, validated artifact produced from a user request. Not a stack layer. | "Intent Layer" (when used to mean the UI) |
| **Intelligence Layer** | The AI cognition/reasoning/routing layer. | AI Cognition (as a layer name) |
| **Agent Runtime** | The environment where Agents execute. | — |
| **Execution Engine** | The kernel component that runs execution plans. | — |
| **Capability** | Executable ability contract + implementation (see Conflict 4). | — |
| **Plugin** | Distribution package that registers Capabilities/Agents (see Conflict 4). | — |
| **Marketplace** | The single distribution channel for Plugins. | Plugin Marketplace, Capability Marketplace |

Note: "Admin Runtime" is **removed** as a stack layer. It conflated the Control Center
(a UI surface) with the Agent Runtime (an execution layer). The canonical stack is:

```
Interaction Layer → Intelligence Layer → Agent Layer → Capability Layer
→ Communication Layer → Runtime Layer → [Security: cross-cutting] → Kernel
```

The Control Center is a surface within the Interaction Layer, not a separate runtime.

## 7. Load-bearing decisions (please confirm)

Most of this RFC is mechanical reconciliation. Two decisions are genuinely architectural
and worth an explicit confirmation before the rest is applied, because everything else
builds on them:

1. **Plugin = distribution package, Capability = execution primitive** (Section 4).
   This is the recommended resolution and matches the existing
   `Plugin → Capability Registration` flow. If you intended Plugin and Capability to be
   independent peer concepts, Sections 3 and 4 change.
2. **Stages are canonical, Phases are workstreams** (Section 2). If you prefer Phases as
   the primary roadmap axis, the mapping table inverts.

The other four conflicts (numbering, repo layout, security ownership, terminology) have
no reasonable alternative and can be applied as-is.

## 8. Rollout

1. Approve this RFC (or amend Sections 2/4).
2. Apply Section 1 renames and delete stale `10.35.NN` headings.
3. Replace the trees in `README.md` and update `ROADMAP.md` to a pointer.
4. Add `Canonical source:` headers per Section 5.
5. Add the Plugin↔Capability paragraph to `05.3.*` and the glossary to Chapter 04.
6. Add a CI check: fail the build if a document number is duplicated, if a deprecated
   alias appears, or if a security doc outside the canonical tree lacks a
   `Canonical source:` header.

Step 6 is what prevents these six conflicts from recurring.

---

## 9. Summary

This RFC gives every contested concept a single owner: one numbering scheme, one roadmap
(two axes), one repository layout, one relationship between Plugin and Capability, one
canonical security model, and one glossary. It changes documentation only, in keeping
with the project's documentation-first rule, and adds a CI guard so the corpus cannot
drift apart again.

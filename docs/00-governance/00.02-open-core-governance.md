# RFC-0003 — Open-Core Governance & Repository Visibility

Version: 1.0
Status: Proposed (awaiting maintainer approval)
Category: Governance
Author: Lead Architect

---

## 0. Decision

cmdOS adopts an **open-core** model.

- **Open ecosystem:** documentation of vision/architecture (contracts), `schemas/`,
  `sdk/`, and example capabilities/plugins are **public** so a community can build on cmdOS.
- **Closed core:** the kernel/runtime execution internals, AI routing logic, business
  strategy, token economics, and all commercial services are **private** — this is the moat.

Every file and directory carries an explicit visibility: **public**, **private**, or
**community**. This RFC defines the classification, the per-file marker, the repository
split, and the guardrails that keep private content out of public history.

---

## 1. Visibility levels

- **public** — visible to everyone. Contracts, SDKs, open docs, examples.
- **community** — public and explicitly open to outside contributions (PRs, plugins).
  A subset of public, marked separately so contributors know where they're invited.
- **private** — restricted to maintainers. Moat, strategy, secrets, commercial infra.

---

## 2. Classification map (by directory)

| Path | Visibility | Rationale |
|------|-----------|-----------|
| `docs/01-vision` … `docs/04-terminology` | public | Trust, recruiting, community alignment |
| `docs/05-architecture` (public contracts, overviews) | public | Integration requires open contracts |
| `docs/05-architecture/**/kernel/core` (internals) | **private** | Execution moat |
| `docs/06-development` | public | Contribution process |
| `docs/07-product` | public | Positioning |
| `docs/08-business` | **private** | Strategy, pricing, competitors |
| `docs/09-roadmap` (public stages) | public | Direction |
| `docs/09-roadmap` (detailed build plans, internal sequencing) | **private** | Execution detail |
| `docs/**/tokenomics`, token design | **private** | Regulatory + competitive sensitivity |
| `schemas/` | public | Ecosystem must consume the contracts |
| `sdk/` | public | Community builds with it |
| `capabilities/examples`, `plugins/` examples | **community** | Invite contributions |
| `capabilities/` (first-party, non-example) | private → open gradually | Ship value first, open when stable |
| `kernel/`, `runtime/`, `ai/router` (code) | **private** | Core competitive logic |
| `providers/` (adapters) | public interface / private keys | Adapter shape open; credentials never |
| `services/` (identity, billing, marketplace backend) | **private** | Commercial infrastructure |
| `infrastructure/`, deployment config | **private** | Ops + attack surface |
| `tests/` | mixed | Follows the visibility of what they test |
| any file containing keys/secrets/tokens/credentials | **private, always** | Never public under any circumstance |

Security note (Kerckhoffs): the *security model* (permission, policy, sandbox, audit —
`docs/05-architecture/6-security`) stays **public**. Only secrets, keys, and specific
hardening configuration are private. Hiding the model is not security.

## 3. Per-file marker

Every source and doc file declares its visibility so the classification is explicit at
the file level, not only inferred from its directory.

Markdown / docs — second line, under the H1:
```markdown
> Visibility: public
```

Code files — top-of-file comment:
```typescript
// Visibility: private
```

Allowed values: `public`, `community`, `private`. When a file's marker and its directory
default disagree, the **more restrictive** wins, and CI flags the mismatch for review.

A machine-readable index lives at `docs/00-governance/00.03-visibility-map.md` (mirrors
Section 2) and is the source of truth a guard script reads.

---

## 4. Repository split

The current single public repo `foundation` mixes public and private content. Open-core
requires two repositories:

- **`cmdos` (public)** — open docs, `schemas/`, `sdk/`, `capabilities/examples`,
  `plugins/` examples, public architecture contracts.
- **`cmdos-core` (private)** — `kernel/`, `runtime/`, `ai/router`, `services/`,
  `docs/08-business`, tokenomics, internal build plans, infrastructure.

Connection: the public repo publishes the **contracts** (`schemas/`). The private core
**implements** them. The core depends on the public schemas as a versioned dependency;
the public side never depends on the private side.

```
cmdos (public)  ──contracts (schemas/)──▶  consumed by
                                           cmdos-core (private) implements them
```

---

## 5. What NEVER goes to a public repo

- API keys, tokens, private keys, `.env` files, credentials of any kind
- Signing keys, deployment secrets, database URIs
- Tokenomics/token-sale material
- Business strategy, pricing models, investor material
- Kernel execution internals classified private above

These are enforced by `.gitignore` and a pre-push secret scan (Section 7).

---

## 6. Migration plan (phased — solo-friendly)

**Important caveat:** anything already pushed to the public `foundation` history is
already public, including in past commits. Truly privatizing it means either history
rewrite (`git filter-repo`) or accepting prior exposure. Classification governs going
forward; decide per item whether past exposure is acceptable.

**Phase A — now (urgent, low effort):**
1. Confirm no secrets exist anywhere in the public repo (run the scan in Section 7).
2. Add `.gitignore` entries for secrets (`.env`, `*.key`, `*.pem`, `secrets/`).
3. Decide on `docs/08-business` and tokenomics: if present in public history and that's
   not acceptable, remove and (optionally) rewrite history now while the repo is small.

**Phase B — when code begins (M0+):**
4. Create the private `cmdos-core` repository.
5. Keep `schemas/` and `sdk/` in the public repo as the contract boundary.
6. Place `kernel/`, `runtime/`, `services/`, private docs in `cmdos-core`.

**Phase C — ongoing:**
7. Every new file gets a visibility marker (Section 3).
8. CI enforces the guardrails (Section 7).

Doing Phase A while the repo is still small is the cheapest moment to fix exposure.

## 7. Guardrails (CI / pre-push)

A guard prevents private content and secrets from entering public history:

- **Secret scan** — reject commits containing key-like patterns (`.env`, `PRIVATE KEY`,
  long hex/base64 blobs, `sk-`, provider key prefixes).
- **Visibility check** — in the public repo, reject any file marked `private` or matching
  a private path in the visibility map.
- **Marker check** — every doc/code file must declare a `Visibility:` marker.

Starter (public repo, `tools/ci/check-visibility.sh`):
```bash
#!/usr/bin/env bash
set -euo pipefail
fail=0

# a) obvious secrets
if git grep -nIE 'BEGIN (RSA |EC |OPENSSH )?PRIVATE KEY|sk-[A-Za-z0-9]{20,}|xox[baprs]-' -- . 2>/dev/null; then
  echo "ERROR: possible secret committed."; fail=1
fi
# b) private-marked files must not be in the public repo
if git grep -nIl 'Visibility:[[:space:]]*private' -- . 2>/dev/null; then
  echo "ERROR: file marked private present in public repo."; fail=1
fi
# c) secret files must be gitignored
for p in .env "*.pem" "*.key"; do
  if git ls-files --error-unmatch $p >/dev/null 2>&1; then
    echo "ERROR: secret file tracked: $p"; fail=1
  fi
done

[ "$fail" -eq 0 ] && echo "check-visibility: OK"
exit $fail
```

---

## 8. Summary

cmdOS is open-core: an open ecosystem (docs, schemas, SDK, examples) around a closed
moat (kernel, execution internals, business, token, services). Visibility is explicit at
every file via a marker and a central map, enforced by a repo split and CI guardrails.
The security *model* stays public; only secrets and internals are private.
